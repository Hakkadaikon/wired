# RFC 5758 — X.509 Additional Algorithms: ECDSA and SHA-2 Signature OIDs

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc5758.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Scope: what this TLS server SDK's certificate stack consumes/produces —
ecdsa-with-SHA256/SHA384 signature-algorithm OID recognition and matching
in certificates. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 6/12 tested, 1 indirect, 5 untested.**

## §2 Hash Functions

- [ ] 5758-001 (§2) The implementation shall recognize the id-sha224 OBJECT
  IDENTIFIER as identifying the SHA-224 hash algorithm.
  - gap: only the composite ecdsa-with-SHA224 signatureAlgorithm OID
    (5758-007) is recognized; no caller in this SDK ever passes the bare
    id-sha224 digest OID on its own (signatureAlgorithm is always the
    composite form), same as the existing id-sha512 precedent.
- [x] 5758-002 (§2) The implementation shall recognize the id-sha256 OBJECT
  IDENTIFIER as identifying the SHA-256 hash algorithm.
  - test: `tests/crypto/sigalgoid_test.c` — `test_sigalgoid_listed`
- [x] 5758-003 (§2) The implementation shall recognize the id-sha384 OBJECT
  IDENTIFIER as identifying the SHA-384 hash algorithm.
  - test: `tests/crypto/sigalgoid_test.c` — `test_sigalgoid_listed`
- [~] 5758-004 (§2) The implementation shall recognize the id-sha512 OBJECT
  IDENTIFIER as identifying the SHA-512 hash algorithm.
  - evidence: `quic_x509_sigalg_lookup` maps the id-sha512-tagged
    sha512WithRSAEncryption OID to `QUIC_X509_HASH_SHA512` and
    `test_sigalgoid_listed` asserts that end-to-end mapping, but no test
    exercises the bare id-sha512 digest-algorithm OID (as opposed to the
    signature-algorithm OID) directly.
- [ ] 5758-005 (§2) When one of the SHA-2 hash OIDs appears in an
  AlgorithmIdentifier, the implementation shall accept both NULL and absent
  parameters as legal and equivalent encodings.
  - gap: `quic_tbscert_sigalg_oid`/`quic_x509_sigalg_lookup` read only the
    OID element and never inspect a trailing parameters element, so NULL
    and absent are indistinguishable to the current code, but no test
    pins a NULL-parameters signatureAlgorithm to demonstrate acceptance.

## Out of scope

Algorithms and CA-generation policy this server SDK does not implement,
excluded from the coverage denominator:

- (§3.1) DSA signatures (id-dsa-with-sha224, id-dsa-with-sha256) — the SDK
  implements only RSA-PKCS1 and ECDSA signature verification/generation; no
  DSA code path exists anywhere in `src/crypto/`.
- (§3.1, §3.2) FIPS 186-4-conformance generation requirements ("Conforming
  CA implementations ... MUST generate ... in accordance with all the
  requirements in Sections 4.1, 4.5, and 4.6 of [FIPS186-3]") — a CA
  operational-policy statement, not a testable wire-format or parsing
  behavior of this server SDK.
- (§5) Security-considerations guidance pointing to NIST SP 800-56A/57/107
  and X9.62 for key-size selection — operator policy guidance, not
  implementable code behavior.
