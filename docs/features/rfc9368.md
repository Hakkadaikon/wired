# RFC 9368 — Compatible Version Negotiation for QUIC

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9368.txt`, not in git), restricted to the server side.
Each requirement carries the test that demonstrates it; an unchecked box
with no test line is an open gap. Status as of 2026-07.

Scope note: this server never actively switches a connection's version —
it replies in whichever version the client's own Initial arrived in, so
compatible version negotiation always degenerates to selecting the
client's Chosen Version (the identity conversion).

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 17/21 tested, 3 indirect, 1 untested.**

## §2 Version Negotiation Mechanism

- [x] 9368-001 (§2) When the server can parse a first flight in the version
  it arrived with, the server shall establish the connection using the
  client's Chosen Version.
  - test: `tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial`
  - test: `tests/app/srvboot_version_test.c` — `test_vneg_not_owed_for_v2`

## §2.1 Incompatible Version Negotiation

- [x] 9368-002 (§2.1) If the server cannot parse the version of a client's
  first flight, then the server shall start incompatible version
  negotiation by sending a Version Negotiation packet.
  - test: `tests/app/srvboot_version_test.c` —
    `test_vneg_owed_for_alien_version`
- [x] 9368-003 (§2.1) The server shall include each entry from its set of
  Offered Versions in the Version Negotiation packet's Supported Version
  fields.
  - test: `tests/app/srvboot_version_test.c` — `test_vneg_lists_v1_and_v2`
- [~] 9368-004 (§2.1) The server shall ignore all received Version
  Negotiation packets.
  - evidence: a Version-0 datagram is excluded both from the server's VN
    trigger (`srvboot_vn_alien` returns 0 for version 0) and from the
    Initial accept gate (`wired_srvboot_is_initial` rejects it); no
    dedicated unit test feeds a VN packet to the server.

## §2.2 Compatible Versions

- [x] 9368-005 (§2.2) The server shall not assume compatibility between two
  versions unless it is explicitly specified (v1 and v2 are compatible;
  unknown pairs are not).
  - test: `tests/transport/compat_test.c` — `test_v1_v2_compatible`
  - test: `tests/transport/compat_test.c` — `test_unknown_incompatible`

## §2.3 Compatible Version Negotiation

- [x] 9368-006 (§2.3) When selecting from the client's Available Versions,
  the server shall select only a version that it supports and that the
  client's Chosen Version is compatible with.
  - test: `tests/transport/verselect_test.c` — `test_verselect_pick`
  - test: `tests/transport/verselect_test.c` — `test_verselect_none`
- [x] 9368-007 (§2.3, §3) The server shall never select a reserved
  (0x?a?a?a?a) version when choosing a version to use.
  - test: `tests/transport/verselect_test.c` — `test_verselect_skips_grease`
  - test: `tests/transport/availfilter_test.c` —
    `test_availfilter_grease_excluded`
- [~] 9368-008 (§2.3) If the server is unable to convert the client's first
  flight into the Negotiated Version, then the server shall abort the
  handshake.
  - evidence: this server only ever selects the arriving version, so the
    conversion is always the identity transformation and cannot fail
    (`tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial` demonstrates the identity path).
- [x] 9368-009 (§2.3) If the server does not find a compatible version
  (including the client's Chosen Version), then the server shall perform
  incompatible version negotiation instead.
  - test: `tests/app/srvboot_version_test.c` —
    `test_vneg_owed_for_alien_version`

## §3 Version Information

- [x] 9368-010 (§3, §10.1) The server shall exchange Version Information as
  the version_information transport parameter (0x11).
  - test: `tests/transport/version_test.c` — `test_version_info_roundtrip`
- [x] 9368-011 (§3) The implementation shall encode and decode Version
  Information as a 32-bit Chosen Version followed by 32-bit Available
  Versions, each in network byte order.
  - test: `tests/transport/verinfo_test.c` — `test_verinfo_roundtrip`
  - test: `tests/transport/verinfo_test.c` — `test_verinfo_wire_be`
- [x] 9368-012 (§3) The implementation shall treat a Version Information
  whose Available Versions field is empty (Chosen Version alone) as
  well-formed, as permitted for server-sent Version Information.
  - test: `tests/transport/verinfo_test.c` — `test_verinfo_chosen_only`

## §4 Version Downgrade Prevention

- [x] 9368-013 (§4) If the peer's Version Information is too short or its
  length is not divisible by four, then the server shall treat it as a
  parsing failure and reject it.
  - test: `tests/transport/verinfo_test.c` — `test_verinfo_decode_bad`
  - test: `tests/transport/version_test.c` — `test_version_info_truncated`
- [ ] 9368-014 (§4) If parsing the peer's Version Information fails, then
  the server shall close the connection with a transport error of type
  TRANSPORT_PARAMETER_ERROR.
  - gap: decode failure is surfaced to the caller, but no test pins the
    resulting close code.
- [~] 9368-015 (§4) If the server receives a Chosen Version equal to zero,
  or any Available Version equal to zero, it shall treat the Version
  Information as a parsing failure.
  - evidence: a zero Chosen Version can never equal the version in use, so
    the Chosen/in-use check fails it (`tests/transport/vneg_test.c` —
    `test_vneg_downgrade_checks`); a zero entry inside Available Versions
    is not explicitly rejected or tested.
- [x] 9368-016 (§4) If the client's Chosen Version is not included in its
  Available Versions, then the server shall treat the Version Information
  as a parsing failure.
  - test: `tests/transport/vneg_test.c` — `test_vneg_downgrade_checks`
- [x] 9368-017 (§4) The server shall validate that the client's Chosen
  Version matches the version in use for the connection, and enter the
  version-negotiation-error state when they differ.
  - test: `tests/transport/vneg_test.c` — `test_vneg_downgrade_checks`
  - test: `tests/transport/verselect_test.c` — `test_verselect_chosen_ok`
- [x] 9368-018 (§4, §10.2) The server shall recognize
  VERSION_NEGOTIATION_ERROR (0x11) as a standard transport error code for
  signaling version negotiation errors.
  - test: `tests/common/codes_test.c` — `test_codes_standard`
- [x] 9368-019 (§4) Where the client's Version Information is absent, the
  server shall still be able to complete the handshake.
  - test: `tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial`

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§1.2, §2.5) Client choice of the Original Version — client behavior.
- (§2.1) Client processing of a Version Negotiation packet (searching the
  list, aborting, selecting a mutual version, sending a new first flight) —
  client behavior.
- (§2.1) "Clients MUST NOT send Version Negotiation packets" — client
  behavior.
- (§2.4) The conceptual two-connection model of incompatible negotiation —
  descriptive, no implementation requirement.
- (§3) Client-sent Available Versions content and preference ordering —
  client behavior.
- (§4) Client-side downgrade validations (ignoring VN packets containing
  the Original Version, validating the server's Available Versions,
  requiring server Chosen Version == Negotiated Version, closing when the
  server's Version Information is missing after a VN reaction) — client
  behavior.
- (§5) The operator three-step rollout for adding or removing a version
  across a server fleet — operator guidance.
- (§7) Design considerations for future QUIC versions (Retry, TLS
  resumption, and 0-RTT interactions of hypothetical compatible versions) —
  guidance to specification authors.
- (§8) Defaulting a missing server version_information to v1 when reacting
  to a Version Negotiation packet — client behavior.
- (§10) IANA registrations — registry process, not implementation.
