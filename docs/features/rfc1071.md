# RFC 1071 — Computing the Internet Checksum

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc1071.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 4/5 tested, 1 indirect, 0 untested.**

This checksum is computed by wired itself only on the AF_XDP path
(`src/transport/io/socket/net/checksum.c`, shared by the IPv4 and UDP
framing); the interop runs used kernel UDP sockets, which compute their own
checksums, so this algorithm is unit-tested but not interop-proven.

## §1 Introduction

- [x] 1071-001 (§1) The implementation shall pair adjacent octets to be
  checksummed into 16-bit integers and form the one's complement sum of
  those integers.
  - test: `tests/transport/net_test.c` — `test_checksum_rfc1071`
- [x] 1071-002 (§1) To generate a checksum, the implementation shall clear
  the checksum field, compute the 16-bit one's complement sum over the
  octets concerned, and place the one's complement of that sum in the
  checksum field.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`
  - test: `tests/transport/net_test.c` — `test_udp_checksum`
- [x] 1071-003 (§1) To check a checksum, the implementation shall compute
  the one's complement sum over the same octets including the checksum
  field, and the check shall succeed only if the result is all one bits.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`
  - test: `tests/transport/net_test.c` — `test_udp_checksum`
- [~] 1071-004 (§1) On a two's complement machine, the implementation shall
  compute the one's complement sum by adding any overflow from the most
  significant bits back into the least significant bits (end-around carry).
  - evidence: `quic_cksum_fold` implements the end-around carry as
    `while (sum >> 16) sum = (sum & 0xffff) + (sum >> 16);`
    (`src/transport/io/socket/net/checksum.c`); exercised by every checksum
    test but with no test isolating the carry step from the fold/complement.
- [x] 1071-005 (§1) The implementation shall pad the octet sequence with a
  zero octet at the end, if necessary, to make it a multiple of two octets
  before summing.
  - test: `tests/transport/net_test.c` — `test_checksum_rfc1071`

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2) Commutative/associative summation order, byte-order independence,
  and parallel summation as ways to speed up the calculation on specific
  CPU word sizes — implementation-technique guidance, not a behavior
  distinguishable at the SDK's API boundary from the plain algorithm.
- (§2) Deferred carries, loop unwinding, combining with data copying, and
  incremental update as performance techniques — optimization guidance, not
  normative requirements.
- (§3) The numerical worked examples for byte-by-byte, normal-order, and
  swapped-order summation — illustrative only; the RFC's own worked example
  in §1/§3 is the one already used as the test vector above.
- (§4) The C, Motorola 68020, Cray, and IBM 370 assembly implementation
  examples — reference implementations for other architectures, not
  requirements on this SDK.
- (Appendix / IEN 45) William Plummer's "TCP Checksum Function Design"
  discussion of checksum properties (P1-P7), product codes, CRC schemes, and
  outboard processing — historical design rationale and alternatives that
  were not adopted, not requirements.
