# FIPS 186-4 — Digital Signature Standard (DSS)

EARS requirement ledger extracted from the spec text
(`tasks/specs/fips186-4.pdf`, not in git), scoped to ECDSA over the
P-256/P-384 prime-field curves as the SDK ships them: P-256 sign
(`src/crypto/asymmetric/ecc/p256sign/sign.c`, deterministic per-message
secret via RFC 6979) and verify
(`src/crypto/asymmetric/ecc/p256/ecdsa_verify.c`), and P-384 verify only
(`src/crypto/asymmetric/ecc/p384/ecdsa_verify.c` — the SDK never signs with
P-384). RSA key generation, plain DSA, ECDSA key-pair generation, and
curves other than P-256/P-384 are not shipped (confirmed by grepping
`src/crypto/asymmetric/`: `rsa/` holds PKCS#1/PSS verify only, no keygen or
DSA anywhere). Each requirement carries the test that demonstrates it; an
unchecked box with no test line is an open gap. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 16/23 tested, 3 indirect, 4 untested.**

  - test: `tests/crypto/p384_point_test.c` — `test_p384_g_on_curve`
- [x] F186-003 (Appendix D.1.2.3) For P-256, the implementation shall use

  - test: `tests/crypto/p256_point_test.c` — `test_p256_g_on_curve`
  - test: `tests/crypto/p256_point_test.c` — `test_p256_scalar`
- [x] F186-004 (Appendix D.1.2.4) For P-384, the implementation shall use

## §6.2 Private/Public Keys

- [~] F186-006 (§6.2) The implementation shall treat an ECDSA key pair as
  usable only for ECDSA digital signature generation and verification, not
  for any other purpose (e.g. key establishment).
  - evidence: the P-256/P-384 sign and verify entry points take raw
    scalar/point byte arrays and are called only from the TLS
    CertificateVerify path (`src/tls/`); no shared code path reuses an
    ECDSA key for key agreement. No dedicated unit test asserts the
    restriction, since it is an API-shape property rather than a runtime
    check.
- [ ] F186-007 (§6.2.1, Appendix B.4) The implementation shall generate an
  ECDSA private key d uniformly in [1, n-1] and compute the corresponding
  public key Q = dG.
  - gap: the SDK never generates ECDSA key pairs; keys are supplied
    externally (loaded from PEM/DER). Out of scope per the file header, but
    listed here rather than dropped because §6.2.1 is a normative
    requirement of the section under otherwise-in-scope Section 6.

## §6.3 Secret Number Generation

- [x] F186-008 (§6.3) The implementation shall generate a new per-message
  secret number k prior to each signature generation, distinct from any
  other signature's k value derived from a different message.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_ps_roundtrip`
- [x] F186-009 (§6.3) The implementation shall compute k^-1 mod n, the
  multiplicative inverse required for signature generation, satisfying
  1 = (k^-1 * k) mod n.
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`
- [~] F186-010 (§6.3) The per-message secret number k and its uses shall be
  protected from unauthorized disclosure; k must not be reused across
  distinct messages under the same key.
  - evidence: k is derived deterministically per-message from
    (private key, message hash) via RFC 6979 HMAC_DRBG
    (`quic_p256sign_k`, see the [RFC 6979 ledger](rfc6979.md)), which by
    construction yields a different k for a different hash input; no
    test directly demonstrates non-disclosure since k is never persisted
    or exposed by the API.

## §6.4 ECDSA Digital Signature Generation and Verification

- [x] F186-011 (§6.4) The implementation shall generate an ECDSA signature
  (r, s) using the domain parameters, the private key, the per-message
  secret number k, and an approved hash function (SHA-256 for P-256).
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`
- [x] F186-012 (§6.4) The implementation shall verify an ECDSA signature
  using the same domain parameters and hash function that were used
  during signature generation.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_valid`
  - test: `tests/crypto/ecdsa_p384_verify_test.c` — `test_ecdsa_p384_valid`
- [~] F186-013 (§6.4) When the hash function's output is longer than the
  bit length of n, the implementation shall use only the left-most n bits
  of the hash output in the signature/verification calculation.
  - evidence: for P-256 (n is 256 bits) and P-384 (n is 384 bits) the
    SDK's shipped hash outputs (SHA-256, SHA-384) are exactly n bits, so
    the "leftmost n bits" truncation is the identity transform and is not
    separately exercised.
  - gap: no test forces a hash output strictly longer than n (e.g.
    SHA-384 with P-256) to demonstrate the truncation branch itself.
- [x] F186-014 (§6.4, referencing ANS X9.62) The implementation shall reject
  a signature verification whose flipped/tampered hash no longer matches
  the signed message.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_bad_hash`
  - test: `tests/crypto/ecdsa_p384_verify_test.c` — `test_ecdsa_p384_bad_hash`
- [x] F186-015 (§6.4, referencing ANS X9.62) The implementation shall reject
  a signature verification whose (r, s) has been tampered with.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_bad_sig`
  - test: `tests/crypto/ecdsa_p384_verify_test.c` — `test_ecdsa_p384_swapped`

## §6.4 Signature validity range checks (ANS X9.62, as implemented)

- [x] F186-016 (§6.4) If r is not in the range [1, n-1], then the
  implementation shall reject the signature as invalid without further
  computation.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_zero_r`
  - test: `tests/crypto/ecdsa_p384_verify_test.c` — `test_ecdsa_p384_zero_r`
  - test: `tests/crypto/p256sign_test.c` —
    `test_p256sign_verify_rejects_zero`
- [ ] F186-017 (§6.4) If s is not in the range [1, n-1], then the
  implementation shall reject the signature as invalid without further
  computation.
  - test: `tests/crypto/p256sign_test.c` —
    `test_p256sign_verify_rejects_zero`
  - gap: that test only exercises s = 0; no test drives s = n or s > n-1
    to force the upper-bound branch of `scalar_in_range`.
- [x] F186-018 (§6.4) The implementation shall reject a public key Q that
  does not lie on the curve.
  - test: `tests/crypto/ecdsa_p384_verify_test.c` —
    `test_ecdsa_p384_off_curve`
  - gap: no equivalent off-curve public-key test exists for P-256
    (`ecdsa_verify_test.c` covers zero-r, bad-hash, and bad-sig only).
- [x] F186-019 (§6.4) The implementation shall compute
  R = u1*G + u2*Q with u1 = e*s^-1 mod n and u2 = r*s^-1 mod n, and accept
  the signature iff R is not the point at infinity and (R.x mod n) == r.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_valid`
  - test: `tests/crypto/ecdsa_p384_verify_test.c` — `test_ecdsa_p384_valid`
- [x] F186-020 (§6.4, low-S convention as applied) The implementation shall
  normalize a generated signature's s component to the lower half of the
  range, replacing s with n-s when s > n/2, so s is always <= n/2.
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_low_s`

## §6.5 Assurances

- [ ] F186-021 (§6.5, §3.1) Prior to generating signatures, the intended
  signatory shall have assurance of the validity of the domain parameters
  and of possession of the private key.
  - gap: operator/provisioning concern (how a private key reaches the
    process); not a property the signing primitive itself can enforce or
    a unit test can demonstrate.
- [ ] F186-022 (§6.5, §3.3) Prior to accepting a signature as valid, the
  verifier shall have assurance of the claimed signatory's identity, the
  domain parameter validity, the public key's validity, and that the
  signatory possessed the private key at signing time.
  - gap: identity/possession assurance is a certificate-chain and PKI
    concern layered above the raw ECDSA verify primitive (see
    `docs/features/rfc5280.md` for the certificate-path side); the raw
    verify function covers only the mathematical check (F186-012,
    F186-016..F186-019).

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§4, Appendix A, Appendix C.3-C.10) DSA domain parameter generation and
  validation (finite-field cryptography), primality testing (Miller-Rabin,
  Lucas, Shawe-Taylor), and the DSA signature/verification algorithm
  itself — the SDK ships no DSA.
- (§5, Appendix B.3, Appendix C.9-C.10) RSA key pair criteria and
  generation (provable and probable primes) — the SDK's `rsa/` directory
  contains PKCS#1 v1.5 and PSS signature *verification* only; it never
  generates or holds an RSA private key.
- (§6.1.1, §6.1.2) ECDSA domain parameter *generation* (as opposed to using
  a fixed NIST curve) and domain parameter management/registration — the
  SDK hard-codes the P-256/P-384 domain parameters from Appendix D rather
  than generating or negotiating its own.
- (§6.2.1, Appendix B.4) ECDSA key pair *generation* (both the
  extra-random-bits and testing-candidates methods) — the SDK loads
  existing key pairs from certificates/PEM; it does not mint new ECDSA
  keys.
- (§6.2.2) Key pair management over a cryptoperiod — operator/PKI
  lifecycle concern, not implemented by a cryptographic primitive.
- (Appendix B.5) The two per-message secret number generation methods
  based on an RBG (extra random bits / testing candidates) — the SDK uses
  the deterministic RFC 6979 construction instead (tracked in
  `docs/features/rfc6979.md`), which FIPS 186-4 §6.3 permits as "an
  approved method."
- (Appendix C.1) Generic modular inverse computation via Fermat's little
  theorem exponentiation — an implementation technique for computing k^-1
  or s^-1; the SDK's specific inverse routines are exercised as part of
  the sign/verify tests already cited (F186-009, F186-019) rather than as
  a standalone requirement.
- (Appendix C.2) Bit-string/integer conversion routines — informative
  plumbing, exercised implicitly by every big-endian encode/decode in the
  cited tests.
- (Appendix D.1.1, D.1.3, D.1.4, D.2.1, D.2.2, D.2.5-D.2.10) Binary-field
  curves (B-xxx), Koblitz curves (K-xxx), P-192, P-224, P-521, and their
  fast-reduction algorithms — curves and fields the SDK does not
  implement (only P-256 and P-384 are shipped).
- (Appendix E) Proof that v = r in the DSA — mathematical justification
  for DSA, out of scope with DSA itself.
- (Appendix F) Calculating the required number of Miller-Rabin rounds —
  supports DSA/RSA prime generation, out of scope with those.
- (Appendix G) References — informative.
- (§1-§3, §4.4, general) Introduction, glossary, general key-management
  guidance (SP 800-57 cross-references), and the overall
  setup/generation/verification workflow figures — informative process
  description layered above the specific algorithm requirements already
  captured under §6.
