## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§5.2) Reestablishing min_rtt at other times in the connection — optional
  (MAY) heuristic, not implemented.
- (§5.3) Subtracting local delays from RTT samples when acknowledgment
  processing is postponed for missing keys — client 0-RTT acknowledgment
  scenario; the server processes acknowledgments immediately.
- (§6.2.2) Reusing a previous connection's smoothed RTT or a
  PATH_CHALLENGE/PATH_RESPONSE delay as the initial RTT — optional (MAY)
  heuristics, not implemented.
- (§6.2.2.1) Client PTO obligations during server address validation
  (arming the PTO with nothing in flight, sending a Handshake packet or a
  padded Initial on PTO) — client behavior.
- (§6.2.3) Sending unacknowledged CRYPTO data ahead of the PTO expiry to
  speed up handshake completion — optional (MAY) acceleration.
- (§6.2.4) Skipping a packet number to elicit a faster acknowledgment, or
  marking in-flight packets lost instead of probing — optional alternatives
  to the implemented probe strategy.
- (§6.3) Resetting congestion control and loss recovery state on receiving a
  Retry packet — client behavior.
- (§6.4) Discarding recovery state for in-flight 0-RTT packets when 0-RTT is
  rejected — client behavior (the server does not send 0-RTT packets).
- (§7) Conformance of an alternative congestion controller to RFC 8085
  §3.1 — algorithm-selection guideline; the SDK's optional CUBIC/BBR modes
  are beyond this spec.
- (§7.2) Recalculating the initial congestion window when the maximum
  datagram size changes — the SDK uses a fixed 1200-byte maximum datagram.
- (§7.4) Ignoring the loss of Handshake, 0-RTT, and 1-RTT packets that
  arrived before the peer had keys — optional (MAY) leniency, not
  implemented.
- (§7.8) Alternative window-update mechanisms after underutilization
  (RFC 7661) — optional (MAY), not implemented.
- (§8.2) Bundling acknowledgments with other frames or padding to resist
  traffic analysis — application/operator mitigation guidance.
- (§8.3) Marking occasional packets with ECN-CE to detect suppression of
  ECN reports — optional detection strategy, not implemented.
