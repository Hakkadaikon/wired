# FIPS 198-1 — The Keyed-Hash Message Authentication Code (HMAC)

EARS requirement ledger extracted from the spec text
(`tasks/specs/fips198-1.pdf`, not in git), scoped to the HMAC-SHA-256
instantiation the SDK ships (`src/crypto/symmetric/hash/hash/hmac.c`; B =
64, L = 32). Each requirement carries the test that demonstrates it; an
unchecked box with no test line is an open gap. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 7/8 tested, 0 indirect, 1 untested.**

## §3 Cryptographic Keys

- [x] F198-001 (§3) When the key K is longer than the hash function's block
  size B, the implementation shall first hash K with H and use the
  resulting L-byte string as K0.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_long_key`

## §4 HMAC Specification

- [x] F198-002 (§4) The implementation shall compute
  MAC(text) = H((K0 XOR opad) || H((K0 XOR ipad) || text)), where ipad is
  the byte x'36' repeated B times and opad is the byte x'5c' repeated B
  times.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`
  - test: `tests/crypto/hmac_test.c` — `test_hmac_long_key`
- [x] F198-003 (§4 Step 1) If the length of K equals B, then the
  implementation shall set K0 = K directly.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`
- [x] F198-004 (§4 Step 2) If the length of K is greater than B, then the
  implementation shall set K0 = H(K) followed by (B-L) zero bytes.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_long_key`
- [x] F198-005 (§4 Step 3) If the length of K is less than B, then the
  implementation shall append zero bytes to K to form the B-byte string K0.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`
- [x] F198-006 (§4 Steps 4-6) The implementation shall XOR K0 with ipad,
  append text, and apply H to the result to compute the inner hash
  H((K0 XOR ipad) || text).
  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`
- [x] F198-007 (§4 Steps 7-9) The implementation shall XOR K0 with opad,
  append the inner hash result, and apply H a second time to compute the
  final MAC.
  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`

## §5 Truncation

- [ ] F198-008 (§5) Where an application truncates the HMAC output, the
  implementation shall use the lambda left-most bits of the HMAC output as
  the MAC.
  - gap: the SDK's `quic_hmac_sha256` always returns the full 32-byte
    output; no caller truncates it and no truncation helper or test exists.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§1) Introduction — informative.
- (§2) Glossary of terms, acronyms, and parameter/symbol definitions —
  informative.
- (§3) Guidance on key security strength and management (SP 800-57) beyond
  the K > B pre-hash rule already captured (F198-001) — operator concern.
- (§6) Implementation note on precomputing and caching the
  (K0 XOR ipad)/(K0 XOR opad) compression-function intermediates — a
  performance optimization explicitly stated to have no effect on
  interoperability; the SDK recomputes both pads per call.
- (§6) Swapping the underlying Approved hash function H for a different one
  — the SDK ships a single fixed instantiation (HMAC-SHA-256); no runtime
  hash-agility surface exists.
- (Appendix A) Differences between FIPS 198 and FIPS 198-1 — historical,
  not a present-tense requirement.
- (Appendix B) References — informative.
