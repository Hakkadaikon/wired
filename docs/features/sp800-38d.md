# NIST SP 800-38D — Recommendation for Block Cipher Modes of Operation:
Galois/Counter Mode (GCM) and GMAC

EARS requirement ledger extracted from the spec text
(`tasks/specs/sp800-38d.pdf`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 14/25 tested, 5 indirect, 6 untested.**

## §5.1 Block Cipher

- [~] S38D-001 (§5.1) The implementation shall use an approved block cipher
  with a 128-bit block size and a key size of at least 128 bits as the
  underlying cipher for GCM.
  - evidence: `quic_gcm_ctx` (`src/crypto/symmetric/aead/gcm/gcm.h`) takes a
    `quic_aes128*`, a FIPS 197 AES-128 (128-bit block, 128-bit key); no
    dedicated test asserts the block/key size beyond the vector tests
    already using AES-128.
- [x] S38D-002 (§5.1) The implementation shall use only the forward cipher
  function of the underlying block cipher within GCM (the inverse direction
  is not required).
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - test: `tests/crypto/gcm_test.c` — `test_gcm_open`
  - evidence: `quic_gcm_seal`/`quic_gcm_open`
    (`src/crypto/symmetric/aead/gcm/gcm.c`) call only
    `quic_aes128_encrypt`, never a decrypt primitive (none exists in this
    SDK — see the FIPS 197 ledger's "Out of scope").

## §5.2 Two GCM Functions

- [x] S38D-003 (§5.2.1.1) The authenticated encryption function shall take
  a plaintext P, additional authenticated data A, and an initialization
  vector IV as input.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - test: `tests/crypto/gcm_test.c` — `test_gcm_open`
- [~] S38D-004 (§5.2.1.1) The implementation shall support IV, plaintext,
  and AAD bit lengths that are each a multiple of 8 (byte strings), subject
  to len(P) <= 2^39-256, len(A) <= 2^64-1, and 1 <= len(IV) <= 2^64-1.
  - evidence: `quic_span` (`common/bytes/span/span.h`) is byte-granular
    throughout, so sub-byte lengths cannot be constructed; the upper bounds
    are unreachable in practice (no test approaches 2^39 bytes of
    plaintext) and are not separately asserted.
- [ ] S38D-005 (§5.2.1.1) The implementation should restrict IV support to
  96 bits, to promote interoperability, efficiency, and simplicity of
  design.
  - gap: this SDK restricts IV support to exactly 96 bits
    (`QUIC_GCM_NONCE` = 12 bytes in
    `src/crypto/symmetric/aead/gcm/gcm.h`), satisfying the recommendation,
    but no test asserts that a non-96-bit IV is rejected or otherwise
    handled — the API simply has no parameter through which one could be
    supplied.
- [x] S38D-006 (§5.2.1.2) The authenticated encryption function shall
  output a ciphertext C of the same bit length as P, and an authentication
  tag T.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
- [~] S38D-007 (§5.2.1.2) The tag length t shall be one of 128, 120, 112,
  104, 96, 64, or 32 bits, fixed per key, and the implementation shall not
  support any other tag length.
  - evidence: `QUIC_GCM_TAG` (`src/crypto/symmetric/aead/gcm/gcm.h`) fixes
    t = 128 unconditionally; other tag lengths are simply unreachable
    (there is no tag-length parameter), which trivially satisfies "shall
    not support any other value" but is not asserted by a dedicated test.
- [x] S38D-008 (§5.2.2) The authenticated decryption function shall take
  IV, C, A, and T as input and shall output either the plaintext P or FAIL,
  where P is returned only if T is the correct tag for (IV, A, C).
  - test: `tests/crypto/gcm_test.c` — `test_gcm_open`

## §6.2 Incrementing Function

- [x] S38D-009 (§6.2) The implementation shall define the s-bit
  incrementing function, inc_s(X), as incrementing the right-most s bits of
  X modulo 2^s while leaving the remaining left-most bits unchanged.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `ctr_inc` (`src/crypto/symmetric/aead/gcm/gcm.c`) increments
    only the low 32 bits of the 16-byte counter block (inc_32), matching
    the requirement; exercised end to end by the NIST vector (a multi-block
    plaintext requires more than one counter increment to match the
    expected ciphertext).

## §6.3 Multiplication Operation on Blocks

- [x] S38D-010 (§6.3) The implementation shall compute the block product
  X . Y over GF(2^128) using the bit-reflected reduction polynomial R =
  {11100001} || 0^120, processing X's bits most-significant first and
  reducing Y by one bit-shift (XORing R on LSB=1) per iteration.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `gf_mul`/`gf_step` (`src/crypto/symmetric/aead/gcm/gcm.c`)
    implement exactly Algorithm 1 (`z ^= v & mask` per bit of `y` tested
    MSB-first, `v` right-shifted with conditional XOR of `0xe1` — the
    reduction constant R's leading byte — on the popped LSB).

## §6.4 GHASH Function

- [x] S38D-011 (§6.4) The implementation shall compute
  GHASH_H(X_1 || ... || X_m) by folding each 128-bit block X_i into an
  accumulator Y (Y_i = (Y_{i-1} XOR X_i) . H, Y_0 = 0^128) and returning
  Y_m.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `ghash_block`/`ghash_bytes`
    (`src/crypto/symmetric/aead/gcm/gcm.c`) implement exactly this
    recurrence, zero-padding the final partial block via `load_block`.

## §6.5 GCTR Function

- [x] S38D-012 (§6.5) The implementation shall compute GCTR_K(ICB, X) by
  encrypting a sequence of counter blocks derived from ICB via inc_32 and
  XORing the resulting keystream against X, truncating the final block's
  keystream to the length of the final partial input block; GCTR of the
  empty string shall return the empty string.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `ctr_xor`/`ctr_chunk`
    (`src/crypto/symmetric/aead/gcm/gcm.c`) implement this; the
    empty-string case is implicit (a zero-length `quic_span` makes the
    `ctr_xor` loop a no-op), not asserted by a dedicated test.

## §7.1 Authenticated Encryption Function

- [x] S38D-013 (§7.1, Step 1-2) The implementation shall compute the hash
  subkey H = CIPH_K(0^128), and, for a 96-bit IV, shall form the
  pre-counter block J0 as IV || 0^31 || 1.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `gcm_setup` (`src/crypto/symmetric/aead/gcm/gcm.c`) computes
    `st->h = E(K, 0^128)` and builds `j0` as exactly `nonce || 00 00 00 01`
    (only the 96-bit-IV branch is implemented; see "Out of scope").
- [x] S38D-014 (§7.1, Step 3) The implementation shall compute the
  ciphertext as C = GCTR_K(inc_32(J0), P).
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `data_ctr` advances the counter to `inc_32(J0)` before
    `ctr_xor` encrypts the plaintext in `quic_gcm_seal`.
- [x] S38D-015 (§7.1, Step 4-6) The implementation shall compute the tag as
  T = MSB_t(GCTR_K(J0, GHASH_H(A || 0^v || C || 0^u ||
  [len(A)]_64 || [len(C)]_64))), where u and v zero-pad A and C to a block
  boundary.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `gcm_tag` (`src/crypto/symmetric/aead/gcm/gcm.c`) GHASHes AAD
    then ciphertext (each implicitly zero-padded to a block by
    `ghash_bytes`/`load_block`) then the 128-bit `[len(A)]64 || [len(C)]64`
    block, then encrypts with `E(K, J0)` — matching the spec exactly since
    t = 128 makes MSB_128 the identity truncation.

## §7.2 Authenticated Decryption Function

- [x] S38D-016 (§7.2, Step 1) If the bit lengths of IV, A, or C are not
  supported, or if len(T) != t, then the implementation shall return FAIL.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_open`
  - evidence: `quic_gcm_open` rejects `ct.n < QUIC_GCM_TAG` (an
    under-length input, the only length precondition expressible through
    this API's fixed-width IV/tag parameters) before computing anything.
- [x] S38D-017 (§7.2, Step 2-7) The implementation shall recompute T' from
  (K, IV, A, C) using the same H/J0/GHASH/GCTR steps as the encryption
  function.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_nist`
  - evidence: `quic_gcm_open` calls the same `gcm_setup`/`gcm_tag` helpers
    used by `quic_gcm_seal`.
- [x] S38D-018 (§7.2, Step 8) If T = T', then the implementation shall
  return the plaintext P; otherwise, it shall return FAIL and shall not
  reveal P.
  - test: `tests/crypto/gcm_test.c` — `test_gcm_open`
  - evidence: `quic_gcm_open` compares tags with `quic_ct_diff16` (constant
    time; `common/bytes/util/ct.h`) and returns 0 without writing `pt` on
    mismatch; `test_gcm_open` asserts both the tag-flip and the AAD-flip
    rejection cases leave the output buffer at its sentinel value.

## §8 Uniqueness Requirement on IVs and Keys

- [ ] S38D-019 (§8) The probability that the authenticated encryption
  function is ever invoked with the same IV and the same key on two or
  more distinct sets of input data shall be no greater than 2^-32.
  - gap: IV uniqueness is a caller/protocol obligation (RFC 9001 derives a
    fresh nonce per packet number, which this SDK's caller is responsible
    for supplying); the GCM primitive itself has no IV-reuse detection or
    enforcement, and no test exercises the consequence of a repeated IV.
- [ ] S38D-020 (§8.1) Any GCM key established among its intended users
  shall, with high probability, be fresh.
- [ ] S38D-021 (§8.2.1, §8.2.2) The implementation shall construct IVs
  using either the deterministic construction (fixed field + invocation
  field) or the RBG-based construction (random field + free field), with a
  single choice fixed per supported IV length for the life of the key.
- [ ] S38D-022 (§8.3) Unless exclusively using 96-bit IVs from the
  deterministic construction, the total number of authenticated encryption
  invocations shall not exceed 2^32 for a given key.
  - gap: this is the RFC 9001 §6.6 AEAD confidentiality limit
    (2^23 for AES-GCM, stricter than SP 800-38D's own 2^32 ceiling); the
    limit is enforced one layer up, not inside `quic_gcm_seal` itself.
    - evidence: `quic_aead_needs_update`/`QUIC_AEAD_LIMIT_AESGCM`
      (`tests/tls/aeadlimit_test.c` — `test_aeadlimit`) trigger a key
      update before this count is reached.

## §9 Practical Considerations for Validating Implementations

- [ ] S38D-023 (§9.1) GCM shall be implemented only within a cryptographic
  module meeting FIPS 140-2, with a generation unit producing IVs per one
  of the §8.2 constructions; a loss of power shall not cause IV repetition.

## §10 Conformance

- [~] S38D-024 (§10) A conforming implementation may replace the specified
  steps of GHASH/GCTR/GCM-AE/GCM-AD with any mathematically equivalent
  procedure that produces the same output for every input.
  - evidence: `gf_mul`'s bit-reflected, LSB-first-shift formulation
    (`src/crypto/symmetric/aead/gcm/gcm.c`) is an equivalent, more
    software-friendly restatement of the MSB-first Algorithm 1; agreement
    with the spec's own construction is demonstrated by the NIST test
    vector rather than by an independent structural proof.

## Appendix B — Authentication Assurance

- [~] S38D-025 (App. B) A conforming implementation should limit the total
  number of blocks of plaintext and AAD protected under a single key
  (recommended limit 2^64) and should monitor/limit unsuccessful
  verification attempts per key.
  - evidence: the RFC 9001 confidentiality limit
    (`tests/tls/aeadlimit_test.c` — `test_aeadlimit`) forces a key update
    at 2^23 invocations, far below the byte-count concern this appendix
    raises; no failed-authentication counter exists (tracked as a gap in
    the RFC 9001 ledger, item 9001-057).

## Out of scope

Requirements this server SDK deliberately does not implement, or that fall
to the caller/protocol/operator rather than this primitive, excluded from
the coverage denominator:

- (§5.2.1, §5.2.2) GMAC (the GCM specialization restricted to
  authentication-only input with no confidential plaintext) — this SDK
  only ever calls GCM with real plaintext (QUIC packet protection); no
  authentication-only entry point is exposed.
- (§7.1 Step 2, §7.2 Step 3, non-96-bit branch) Deriving J0 via
  GHASH_H(IV || 0^(s+64) || [len(IV)]_64) for IV lengths other than 96
  bits — `QUIC_GCM_NONCE` fixes the nonce at exactly 12 bytes
  (`src/crypto/symmetric/aead/gcm/gcm.h`), so only the 96-bit branch of
  Step 2 is reachable; QUIC (RFC 9001 5.3) always uses a 96-bit AEAD nonce.
- (§8, §8.1, §8.2, §8.3) IV/key uniqueness, key freshness, IV construction
  frameworks, and invocation-count ceilings — these bind the calling
  protocol (RFC 9001 derives the per-packet nonce deterministically from a
  connection secret and the packet number, guaranteeing uniqueness per
  key), not the GCM primitive itself, which has no notion of a "connection"
  or session across which to track reuse.
- (§9, §9.1, §9.2) FIPS 140-2 cryptographic-module validation, IV
  generation-unit hardware/firmware boundaries, and power-loss recovery —
  operator/deployment and hardware-validation concerns, not implementation
  behavior this software SDK can express.
- (Appendix A) Narrative explanation of why IV reuse is dangerous
  (informative, no normative content beyond §8's own requirement, already
  captured as S38D-019).
- (Appendix C) Short-tag (32-/64-bit) usage guidelines and their
  packet-length/invocation-count tables — this SDK only ever uses the
  full 128-bit tag (`QUIC_GCM_TAG` = 16 bytes); the short-tag guidance
  does not apply.
- (Appendix D) Replay protection via IV-duplication monitoring or AAD
  sequence numbers — an application/protocol-layer concern (RFC 9000's
  packet number space provides this for QUIC), not a property of the AEAD
  primitive.
- (Appendix E) Bibliography — reference list, not a requirement.
