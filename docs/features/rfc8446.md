# RFC 8446 — The Transport Layer Security (TLS) Protocol Version 1.3

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8446.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Scope: the server side of TLS 1.3 as embedded in QUIC (RFC 9001) — the
record layer is replaced by QUIC packet protection, so §5 and the other
QUIC-superseded mechanisms are listed under Out of scope.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 70/105 tested, 18 indirect, 17 untested.**

## §4 Handshake Protocol

- [x] 8446-001 (§4) The server shall process handshake messages only in the
  order defined by the handshake protocol.
  - test: `tests/tls/hsdriver_test.c` — `test_out_of_order_rejected`
  - test: `tests/tls/server_test.c` — `test_server_fin_before_flight`
- [~] 8446-002 (§4) If a handshake message is received in an unexpected
  order, then the server shall abort the handshake with an
  "unexpected_message" alert.
  - evidence: out-of-order messages are rejected
    (`test_out_of_order_rejected`) and TLS alerts map onto QUIC CRYPTO_ERROR
    codes (`test_error_crypto`); no test asserts the specific alert code on
    this path.

## §4.1.1 Cryptographic Negotiation

- [x] 8446-003 (§4.1.1) If there is no overlap between the client's offered
  cipher suites and the suites the server supports, then the server shall
  abort the handshake.

  - test: `tests/tls/cipher_test.c` — `test_cipher_select_none`
- [~] 8446-004 (§4.1.1) If the client's "key_share" extension contains no
  share for a group the server supports, then the server shall not derive
  handshake keys or send its flight.
  - evidence: the key_share scan reports absence
    (`test_ext_key_share_scan_absent` in `tests/tls/ext_keyshare_test.c`)
    and the flight builder requires a client share; the abort path itself
    has no dedicated test.
- [~] 8446-005 (§4.1.1) When the server selects a PSK, the server shall
  select a key exchange mode from the client's "psk_key_exchange_modes"
  extension (this SDK requires psk_dhe_ke).

    are tested (`test_pskmodes_roundtrip` in `tests/tls/pskmodes_test.c`);
    the check is not asserted inside the PSK acceptance path.
- [ ] 8446-006 (§4.1.1) When the server selects an (EC)DHE group for which
  the initial ClientHello carries no compatible key share, the server shall
  respond with a HelloRetryRequest.
  - gap: the HRR builder exists and is unit-tested, but no server receive
    path sends it.

## Out of scope

Client-only, operator-level, or QUIC-superseded requirements this server
SDK does not implement, excluded from the coverage denominator:

- (§5, §5.1–§5.5) The entire TLS record layer — framing, TLSCiphertext
  protection, per-record nonces, record padding, key-usage limits — QUIC
  replaces the record layer with QUIC packet protection (RFC 9001 §4;
  AEAD confidentiality limits are enforced per RFC 9001 §6.6 instead).
- (§4.1.3, Appendix D.1–D.2, D.5) Negotiating TLS 1.2 and below, the
  downgrade sentinels in ServerHello.random, and SSL/legacy-version
  handling — QUIC mandates TLS 1.3 or newer (RFC 9001 §4.2).
- (§4.1.2, §4.1.4, §4.2.1, §4.2.8, §4.2.11) Client-side ClientHello
  construction and client-side checking of ServerHello /
  HelloRetryRequest / selected_identity — client behavior.
- (§4.2.2) Ignoring unprotected change_cipher_spec records between
  ClientHellos — a record-layer artifact that cannot occur over QUIC's
  CRYPTO streams.
- (§4.2.4) certificate_authorities-guided certificate selection — the
  server serves one operator-configured chain.
- (§4.2.5) The "oid_filters" extension — client-certificate filtering is
  part of client authentication, which this SDK does not ship.
- (§4.2.6, §4.6.2) post_handshake_auth and post-handshake client
  authentication — QUIC prohibits post-handshake CertificateRequest
  (RFC 9001 §4.4) and the SDK ships no client authentication.
- (§4.2.8.1, §7.4.1) Finite field Diffie-Hellman groups and their public
  key validation — no FFDHE group is offered or selectable.
- (§4.3.2, §4.4.2.3, §4.4.2.4) Requesting client certificates and
  validating a received client Certificate — the server never requests
  client authentication (the CertificateRequest codec exists unwired).
- (§4.4.2.1) OCSP status_request and signed_certificate_timestamp
  CertificateEntry extensions — not shipped.
- (§4.5) EndOfEarlyData — QUIC does not use it; its receipt is a QUIC
  protocol violation (RFC 9001 §8.3).
- (§4.6.3, §7.2) The TLS KeyUpdate message and "traffic upd" secret
  update — QUIC forbids TLS KeyUpdate and rekeys via its own Key Phase
  mechanism with the "quic ku" label (RFC 9001 §6).
- (§6.1) The close_notify / user_canceled closure alerts — QUIC endpoints
  do not use closure alerts (RFC 9001 §4.8).
- (§7.3) [sender]_write_key / [sender]_write_iv record traffic keys —
  QUIC derives packet-protection keys with its own "quic key" / "quic iv"
  / "quic hp" labels (RFC 9001 §5.1).
- (§7.5) Keying material exporters — no exporter interface is required by
  or shipped for QUIC.
- (§8) The client-side duty to send only replay-safe early data — client
  behavior.
- (§9.3) Middlebox protocol invariants — requirements on middleboxes, not
  on a server endpoint.
- (Appendix A.1) The client state machine — client behavior (the SDK's
  test client is covered by its own loopback tests, not this ledger).
- (Appendix D.4) Middlebox compatibility mode (compat session IDs, dummy
  change_cipher_spec records) — QUIC forbids middlebox compatibility mode
  (RFC 9001 §8.4).
- (§2.2) Externally provisioned PSKs and the "ext binder" label — the SDK
  ships resumption PSKs only ("res binder").
