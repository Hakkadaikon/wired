# RFC 9204 — QPACK: Field Compression for HTTP/3

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9204.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
The server advertises `SETTINGS_QPACK_MAX_TABLE_CAPACITY 0` and
`SETTINGS_QPACK_BLOCKED_STREAMS 0`, so several decoder-side dynamic-table
obligations are indirect (`[~]`): the tested configuration makes their
trigger condition unreachable. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 31/55 tested, 12 indirect, 12 untested.**

## §2 Compression Process Overview

- [x] 9204-001 (§2.1) The server shall emit field representations in the
  order the field lines appear in the input field section.
  - test: `tests/app/request_headers_test.c` — `test_get_field_section`
- [~] 9204-002 (§2.1.2) If the server's decoder encounters more blocked
  streams than the promised SETTINGS_QPACK_BLOCKED_STREAMS, then the server
  shall treat this as a connection error of type QPACK_DECOMPRESSION_FAILED.
  - evidence: the server advertises QPACK_MAX_TABLE_CAPACITY 0 and
    QPACK_BLOCKED_STREAMS 0 (`tests/app/h3settings_build_test.c` —
    `test_h3settings_build`), so no stream can ever block; no dedicated test.
- [~] 9204-003 (§2.2) The server shall emit field lines in the order their
  representations appear in the encoded field section.
  - evidence: the request decoder walks representations strictly
    sequentially (`scan_lines` in `request_drive.c`), exercised by
    `test_reqdrive_curl_get`; no dedicated order-assertion test.
- [~] 9204-004 (§2.2.1) When the Required Insert Count of a received field
  section is less than or equal to the server's Insert Count, the server
  shall process the field section immediately.
  - evidence: with the advertised capacity 0 every conformant section
    carries Required Insert Count 0 and is decoded immediately after the
    prefix; exercised by every request-decode test (e.g.
    `test_reqdrive_curl_get`); no dedicated test.
- [~] 9204-005 (§2.2.1) If a received Required Insert Count is smaller than
  the lowest value with which the field section could be decoded, then the
  server shall treat this as a connection error of type
  QPACK_DECOMPRESSION_FAILED.
  - test: `tests/app/insertcount_test.c` —
    `test_qpack_ric_min_below_expected_rejected`
  - gap: `quic_qpack_ric_min_ok` implements and tests the check, but no
    live decode path calls it yet.
- [~] 9204-006 (§2.2.2.1) When the server finishes decoding a field section
  that contains dynamic table references, the server shall emit a Section
  Acknowledgment instruction.
  - evidence: unreachable in the tested configuration (capacity 0 means no
    incoming section may reference the dynamic table); the instruction
    encoder itself round-trips in
    `tests/app/qpack_instruction_test.c` — `test_qpack_dec_instr_roundtrip`.
- [~] 9204-007 (§2.2.2.2) When a stream is reset or reading is abandoned,
  the server shall generate a Stream Cancellation instruction.
  - evidence: the spec exempts decoders whose maximum dynamic table capacity
    is zero (they MAY omit Stream Cancellations); the server advertises 0
    (`tests/app/h3settings_build_test.c` — `test_h3settings_build`).
- [x] 9204-008 (§2.2.3) If a field line representation references a dynamic
  table entry that has been evicted or whose absolute index is greater than
  or equal to the declared Required Insert Count, then the server shall fail
  decoding of the field section.
  - test: `tests/app/field_decode_test.c` — `test_dynamic_miss`
  - test: `fuzz/fuzz_qpack.c` — `LLVMFuzzerTestOneInput`
  - gap: the failure is a generic decode rejection; the
    QPACK_DECOMPRESSION_FAILED connection-error code is not mapped (see
    9204-054).
- [~] 9204-009 (§2.2.3) If an encoder instruction references a dynamic table
  entry that has been evicted, then the server shall treat this as a
  connection error of type QPACK_ENCODER_STREAM_ERROR.
  - test: `tests/app/dynget_test.c` — `test_get_enc_rel_evicted`
  - gap: `quic_qpack_dyn_get_enc_rel` implements and tests the check
    (evicted reference resolves to nothing), but no runtime consumer of
    encoder-stream instructions calls it yet.

## §3 Reference Tables

  - test: `tests/app/field_decode_test.c` — `test_static_reference`

  section.
  - evidence: `quic_qpack_static_get` rejects out-of-range indices

    field-line decoders fail on that rejection; no dedicated field-line test.
- [~] 9204-012 (§3.2) The implementation shall not treat duplicate dynamic
  table entries as an error.
  - evidence: insertion imposes no uniqueness — same-name entries coexist in
    `tests/app/dynfind_test.c` — `test_find_prefers_full`; no dedicated
    name-and-value duplicate test.
- [x] 9204-013 (§3.2.1) The implementation shall account the size of a
  dynamic table entry as its name length plus its value length plus 32.
  - test: `tests/app/dyntable_test.c` — `test_insert_size`
- [x] 9204-014 (§3.2.2) When a new entry does not fit, the implementation
  shall evict entries from the end of the dynamic table until the table size
  is at most the capacity minus the new entry's size.
  - test: `tests/app/dyntable_test.c` — `test_evict_on_overflow`
  - test: `tests/app/dynget_test.c` — `test_get_evicted`
- [x] 9204-015 (§3.2.2) If an entry larger than the dynamic table capacity
  is added, then the implementation shall reject the insertion and leave the
  table unchanged.
  - test: `tests/app/dyntable_test.c` — `test_too_big_rejected`
- [~] 9204-016 (§3.2.3) While the peer's maximum dynamic table capacity is
  zero, the server shall not insert dynamic table entries and shall not send
  encoder instructions on the encoder stream.
  - evidence: the server opens only the control unidirectional stream and
    its response encoders reference the static table or emit literals only
    (`tests/app/field_encode_test.c` — `test_field_encode_status_200`,
    `tests/app/h3settings_control_settings_test.c` —
    `test_h3settings_control_settings`); no dedicated absence test.
- [~] 9204-017 (§3.2.3) While a client resumes with 0-RTT and the remembered
  SETTINGS_QPACK_MAX_TABLE_CAPACITY is non-zero, the server shall send the
  same non-zero value in its SETTINGS frame.
  - evidence: the server always advertises capacity 0
    (`tests/app/h3settings_build_test.c` — `test_h3settings_build`), so a
    remembered non-zero value can never exist.
- [x] 9204-018 (§3.2.4) The implementation shall give the first inserted
  entry absolute index 0 and increase the index by one per insertion, fixed
  for the entry's lifetime.
  - test: `tests/app/dynget_test.c` — `test_get_live`
  - test: `tests/app/dynget_test.c` — `test_get_evicted`
- [~] 9204-019 (§3.2.5) When interpreting encoder instructions, the

  - evidence: the rel-to-abs mapping is tested
    (`tests/app/relindex_test.c` — `test_qpack_rel_to_abs`) but no
    encoder-instruction consumer applies it with the insert count as base.
- [x] 9204-020 (§3.2.5) When interpreting field line representations, the

  index Base minus 1.
  - test: `tests/app/relindex_test.c` — `test_qpack_rel_to_abs`
  - test: `tests/app/field_decode_test.c` — `test_dynamic_roundtrip`
- [x] 9204-021 (§3.2.6) The implementation shall resolve post-Base index 0

  - test: `tests/app/relindex_test.c` — `test_qpack_postbase_to_abs`

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2.1.1) Encoder-side tracking of unacknowledged references to decide
  evictability — the SDK's encoder never inserts dynamic table entries.
- (§2.1.1.1) Draining-index heuristic for avoiding prohibited insertions —
  encoder preference guidance, not shipped.
- (§2.1.2) Encoder budget for streams at risk of blocking — the SDK's
  encoder emits only Required-Insert-Count-0 sections, which cannot block.
- (§2.1.3) Withholding instructions until flow-control credit covers them —
  the server sends no QPACK instructions.
- (§2.1.4) Known Received Count tracking — no dynamic references exist to
  acknowledge.
- (§2.2.1) Retaining blocked-stream data in the flow-control window and
  unblocking on Insert Count growth — the decoder never blocks (dynamic
  table disabled by the capacity-0 setting).
- (§2.2.2.3) Timing and coalescing of Insert Count Increment emission —
  decoder-side dynamic table is disabled.
- (§4.5.4, §7.1.3) Re-encoding rules for never-indexed literals at
  intermediaries — wired is an origin server, not an intermediary.
- (§7.1) Dynamic-table probing (CRIME-style) mitigation strategies — design
  guidance for shared-table deployments; the server never indexes
  peer-supplied values.
- (§7.2) Static Huffman information-leakage discussion — informational.
- (§7.3) Memory-consumption tuning guidance — operator guidance (reflected
  only insofar as the capacity-0 default bounds decoder state).
- (Appendix C) Sample single-pass encoding algorithm — informative example.
