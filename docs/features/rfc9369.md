# RFC 9369 — QUIC Version 2

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9369.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Scope note: this server never actively switches a connection's version —
a v2 Initial is answered in v2 and a v1 Initial in v1 (RFC 9368 §2), so
the §4.1 switching requirements hold with negotiated version == original
version.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 18/25 tested, 4 indirect, 3 untested.**

## §3 Differences with QUIC Version 1

- [~] 9369-001 (§3) Except for the differences listed in this section, the
  server shall implement the QUIC version 1 specification (RFC 9000,
  RFC 9001, RFC 9002) for version 2 connections.
  - evidence: the v2 accept path runs through the same machinery the
    RFC 9000/9001/9002 ledgers cover
    (`tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial` drives a v2 Initial end to end
    through the shared code path); no separate v2 re-run of those suites.

## §3.1 Version Field

- [x] 9369-002 (§3.1) The implementation shall use 0x6b3343cf as the
  version 2 long-header Version field value.
  - test: `tests/transport/verinfo_test.c` — `test_verinfo_wire_be`
  - test: `tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial`

## §3.2 Long Header Packet Types

- [x] 9369-003 (§3.2) The implementation shall map version 2 long-header
  Type field values as Initial=0b01, 0-RTT=0b10, Handshake=0b11,
  Retry=0b00, round-tripping against the v1 mapping.
  - test: `tests/transport/v2types_test.c` — `test_v2_wire_values`
  - test: `tests/transport/v2types_test.c` — `test_roundtrip`
- [x] 9369-004 (§3.2) When parsing a long header, the server shall decode
  the type bits under the mapping selected by the packet's Version field,
  never under the other version's mapping.
  - test: `tests/app/srvboot_version_test.c` — `test_is_initial_v2_recognized`
  - test: `tests/app/srvboot_version_test.c` —
    `test_is_initial_v1_0rtt_bits_rejected`
  - test: `tests/app/srvboot_version_test.c` —
    `test_initpkt_ver_v2_byte0_type_bits`
- [x] 9369-005 (§3.2) If a logical packet type or wire type value is out of
  range, then the implementation shall reject the mapping.
  - test: `tests/transport/v2types_test.c` — `test_invalid`

## §4.1 Compatible Negotiation Requirements

- [~] 9369-014 (§4.1) If the server sends a Retry packet, it shall use the
  original version.
  - evidence: the server never switches versions, so a Retry is always
    built in the arriving (original) version; no version-switch Retry
    test exists.
- [x] 9369-015 (§4.1, §3.3.3) When the version in use changes, the Retry
  integrity tag shall be re-derived under the new version's key material;
  an unchanged version keeps the tag valid.
  - test: `tests/transport/switchrule_test.c` — `test_retry_reencode_on_change`
  - test: `tests/transport/switchrule_test.c` — `test_retry_no_reencode_same`
- [x] 9369-016 (§4.1) The server shall send all CRYPTO frames using the
  negotiated version.
  - test: `tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial`
- [x] 9369-017 (§4.1) The server shall keep 0-RTT keys usable across a
  compatible version switch (accepting original-version 0-RTT packets) and
  drop them on an incompatible switch.
  - test: `tests/transport/switchrule_test.c` — `test_0rtt_keep_compatible`
  - test: `tests/transport/switchrule_test.c` — `test_0rtt_drop_incompatible`
- [~] 9369-018 (§4.1) The server shall not discard its original-version
  Initial receive keys until it successfully processes a Handshake packet
  with the negotiated version.
  - evidence: no version switch ever occurs, so Initial keys follow the
    normal RFC 9001 discard schedule (see the RFC 9001 ledger); no
    switch-specific test.
- [~] 9369-019 (§4.1) The server shall send Handshake and 1-RTT packets
  using the negotiated version and drop such packets using any other
  version.
  - evidence: identity negotiation keeps the whole connection in one
    version (`tests/app/srvboot_version_test.c` —
    `test_srvboot_accept_v2_initial`), and the accept path opens coalesced
    slices only under the arrival version; no cross-version drop test.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§1, §6) Ossification rationale and middlebox-behavior discussion —
  informational.
- (§4) Alt-Svc discovery considerations — HTTP client and operator
  concern.
- (§4.1) Client-side rules: ignoring Retry packets in other versions, not
  switching versions in the token-bearing Initial, learning the negotiated
  version from the long-header Version field, sending subsequent Initials
  in it, and never sending 0-RTT in the negotiated version — client
  behavior.
- (§4.1) Encoding the QUIC version in the Retry token to detect a client
  version switch — optional (MAY); moot while the server never switches
  versions.
- (§5) Client prohibition on using tickets or tokens across versions —
  client behavior.
- (§6) Client choice to initiate a connection with version 2 — client
  behavior.
- (§8) Security considerations — informational; no new requirements over
  version 1.
- (§9) IANA registrations (0x6b3343cf permanent, 0x709a50c4 provisional) —
  registry process, not implementation.
