# RFC 9110 — HTTP Semantics

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9110.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
RFC 9110 is an omnibus spec; this ledger covers the slice a wired HTTP/3
origin server surfaces (methods, status codes, field handling at the
server-loop level) — the rest is grouped under Out of scope.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 21/28 tested, 3 indirect, 4 untested.**

## Out of scope

Upper-layer, client-only, intermediary, or operator requirements this
HTTP/3 server SDK does not implement, excluded from the coverage
denominator:

- (§1–§3) Introduction, conformance prose, and terminology — definitional,
  no server behavior to test.
- (§4) URI schemes, normalization/comparison, and https certificate
  verification — URI processing is the application's; certificate
  verification is client behavior (the server's TLS side is covered by the
  TLS/PKI ledgers).
- (§5.2–§5.3, §5.6) Combined field values, field ordering, and the common
  ABNF rules for defining field values — field values are exposed verbatim
  to the application handler.
- (§6.5–§6.6) Trailer fields and message metadata (Date, Trailer) —
  trailers are not processed; the freestanding SDK has no wall clock to
  originate Date.
- (§7.3–§7.8) Routing to caches/proxies, misdirected-request rejection,
  response correlation, forwarding, Connection, Max-Forwards, Via, message
  transformation, and Upgrade — intermediary behavior; this SDK is an
  origin server only, and HTTP/3 has no Upgrade.
- (§8.4–§8.5, §8.7–§8.8) Content-Encoding, Content-Language,
  Content-Location, and validator fields (Last-Modified, ETag) —
  representation metadata generated by the application handler.
- (§9.2) Safe/idempotent method properties and caching interactions —
  semantic guidance binding the application, not the transport SDK.
- (§9.3.4, §9.3.5, §9.3.8) PUT, DELETE, and TRACE semantics — optional
  methods routed to the application handler; TRACE is deliberately not
  shipped.
- (§10.1.2–§10.2) Request/response context fields (From, Referer, TE,
  User-Agent, Allow, Location, Retry-After, Server) — generated and
  consumed by the application.
- (§11) HTTP authentication — application concern.
- (§12) Content negotiation — application concern.
- (§13) Conditional requests — require validators and caching state the
  SDK does not ship.
- (§14) Range requests — application/static-file feature not shipped.
- (§15.2–§15.6, except the codes itemized above) Individual status code
  definitions — the status sent is chosen by the application handler; the
  SDK only encodes it.
- (§16) Extension registries and considerations for new methods, codes,
  fields, and schemes — IANA/author guidance.
- (§17–§18) Security and IANA considerations — informational guidance with
  no directly testable server behavior.
