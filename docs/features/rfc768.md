# RFC 768 — User Datagram Protocol

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc768.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 3/6 tested, 1 indirect, 2 untested.**

These headers are built by wired itself only on the AF_XDP path
(`src/transport/io/socket/net/udp4.c`); the interop runs used kernel UDP
sockets, which fill in the UDP header themselves, so this framing is
unit-tested but not interop-proven.

## Format

- [x] 768-001 (Format) The implementation shall lay out the UDP header as
  Source Port, Destination Port, Length, and Checksum, each a 16-bit field,
  followed by the data octets.
  - test: `tests/transport/net_test.c` — `test_udp_checksum`
- [ ] 768-002 (Fields) The implementation shall insert a value of zero for
  Source Port when no meaningful source port is available.
- [x] 768-003 (Fields) The implementation shall set Length to the length in
  octets of the UDP header plus the data, with a minimum value of eight.
  - test: `tests/transport/net_test.c` — `test_udp_checksum`
- [x] 768-004 (Fields) The implementation shall compute Checksum as the
  16-bit one's complement of the one's complement sum of the pseudo-header
  (source address, destination address, protocol, UDP length), the UDP
  header, and the data, padded with a zero octet if the data is an odd
  number of octets.
  - test: `tests/transport/net_test.c` — `test_udp_checksum`
  - test: `tests/transport/net_test.c` — `test_net_datagram_over_link`
- [~] 768-005 (Fields) If the computed checksum is zero, then the
  implementation shall transmit it as all ones.
  - evidence: the checksum is produced by `quic_cksum_fold`, whose
    end-around-carry-then-complement construction can output the all-ones
    field, but no test drives an input whose computed sum yields this exact
    boundary value.
- [ ] 768-006 (Fields) Where the sender generated no checksum, the
  implementation shall transmit an all-zero checksum field to indicate this
  to the receiver.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (User Interface) The receive-port creation, receive, and send operations
  described for a UDP user interface — an application/operator-facing API
  shape, not a wire-format requirement.
- (IP Interface) The UDP module determining source/destination addresses and
  protocol from the internet header via a full-datagram IP interface —
  covered by the IP layer itself (see the RFC 791 ledger), not a distinct
  UDP requirement.
- (Protocol Application) The Internet Name Server and Trivial File Transfer
  as example UDP applications — informational, not a requirement on this
  SDK.
- (Protocol Number) UDP being assigned protocol number 17 — a registry fact
  reflected as the `QUIC_IP_PROTO_UDP` constant, not a testable behavior in
  its own right.
