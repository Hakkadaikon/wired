# RFC 8439 — ChaCha20 and Poly1305 for IETF Protocols

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8439.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 17/21 tested, 4 indirect, 0 untested.**

## §2.1 The ChaCha Quarter Round

- [x] 8439-001 (§2.1) The implementation shall compute the quarter round on
  four 32-bit words as the fixed add/XOR/roll sequence (rolls of 16, 12, 8,
  and 7 bits) with addition modulo 2^32.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_block`

## §2.3 The ChaCha20 Block Function

- [x] 8439-002 (§2.3) The implementation shall initialize state words 0-3
  with the constants 0x61707865, 0x3320646e, 0x79622d32, 0x6b206574.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_block`
- [x] 8439-003 (§2.3) The implementation shall load the 256-bit key into
  words 4-11 and the 96-bit nonce into words 13-15 as little-endian 32-bit
  integers, with the block counter in word 12.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_block`
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_encrypt`
- [x] 8439-004 (§2.3) The implementation shall run 20 rounds as 10
  iterations of the four column quarter rounds followed by the four diagonal
  quarter rounds.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_block`
- [x] 8439-005 (§2.3) When the rounds complete, the implementation shall add
  the original state words modulo 2^32 and serialize the sixteen words
  little-endian into the 64-byte block.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_block`
- [~] 8439-006 (§2.3, §2.8) The implementation shall not repeat a nonce for
  the same key.
  - evidence: in this SDK the AEAD nonce is the QUIC packet-protection IV
    XORed with the strictly increasing packet number (RFC 9001 §5.3, see the
    [RFC 9001 ledger](rfc9001.md)); no dedicated uniqueness test at this
    layer.

## §2.4 The ChaCha20 Encryption Algorithm

- [x] 8439-007 (§2.4) The implementation shall build the keystream by
  calling the block function with the same key and nonce and successively
  incrementing block counters, XORing it with the plaintext.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_encrypt`
- [x] 8439-008 (§2.4) When the plaintext is not a multiple of 64 bytes, the
  implementation shall use only the needed prefix of the final keystream
  block and discard the remainder.
  - test: `tests/crypto/chacha20_test.c` — `test_chacha_encrypt`
- [x] 8439-009 (§2.4) The implementation shall decrypt by XORing the same
  keystream with the ciphertext, recovering the plaintext.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_open`

## §2.5 The Poly1305 Algorithm

- [x] 8439-010 (§2.5) The implementation shall clamp r before use: the top
  four bits of r[3], r[7], r[11], r[15] cleared and the bottom two bits of
  r[4], r[8], r[12] cleared.
  - test: `tests/crypto/poly1305_test.c` — `test_poly1305_rfc`
- [x] 8439-011 (§2.5) The implementation shall process the message in
  16-byte blocks read as little-endian numbers with one bit appended beyond
  the final octet, computing Acc = ((Acc + block) * r) mod 2^130-5.
  - test: `tests/crypto/poly1305_test.c` — `test_poly1305_rfc`
- [x] 8439-012 (§2.5) When all blocks are absorbed, the implementation shall
  add s to the accumulator and serialize the least significant 128 bits
  little-endian as the tag.
  - test: `tests/crypto/poly1305_test.c` — `test_poly1305_rfc`
- [~] 8439-013 (§2.5) The implementation shall use a (r,s) pair that is
  unpredictable for each invocation.
  - evidence: the one-time key is derived per nonce from the session key via
    the §2.6 procedure (8439-014); unpredictability itself is not
    demonstrable by a unit test.

## §2.6 Generating the Poly1305 Key Using ChaCha20

- [x] 8439-014 (§2.6) The implementation shall generate the one-time
  Poly1305 key as the first 32 bytes of the ChaCha20 block computed with the
  session key, the nonce, and block counter zero.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_rfc`

## §2.8 AEAD Construction

- [x] 8439-015 (§2.8) The implementation shall encrypt the plaintext with
  ChaCha20 using the AEAD key and nonce and the initial counter set to 1.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_rfc`
- [x] 8439-016 (§2.8) The implementation shall compute the tag with Poly1305
  over AAD || pad16(AAD) || ciphertext || pad16(ciphertext) ||
  le64(len(AAD)) || le64(len(ciphertext)).
  - test: `tests/crypto/aead_test.c` — `test_chapoly_rfc`
- [x] 8439-017 (§2.8) The implementation shall output the ciphertext of the
  same length as the plaintext followed by the 128-bit tag.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_rfc`
- [x] 8439-018 (§2.8) When decrypting, the implementation shall run Poly1305
  over the AAD and the ciphertext (not the plaintext) and accept the message
  if and only if the computed tag matches the received tag.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_open`
- [x] 8439-019 (§2.8) If the received tag does not match, then the
  implementation shall reject the message without releasing the plaintext.
  - test: `tests/crypto/aead_test.c` — `test_chapoly_open`

## §4 Security Considerations

- [~] 8439-020 (§4) The implementation shall compare the calculated and
  received tags with a constant-time comparison function.
  - evidence: `quic_chapoly_open` compares via the shared constant-time
    helper `quic_ct_diff16` (`common/bytes/util/ct.h`); constant-time
    behavior is not demonstrable by a unit test.
- [~] 8439-021 (§4) The implementation shall include the complete tag and
  shall not truncate it.
  - evidence: the seal/open APIs carry a fixed 16-byte tag
    (`QUIC_CHAPOLY_TAG`); every AEAD test round-trips the full tag, and no
    API accepts a shorter one.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2.3) 128-bit keys and the 8/12-round ChaCha variants — excluded by the
  spec itself.
- (§2.5) generating the one-time key by encrypting a nonce with AES —
  informational context; this SDK uses the §2.6 ChaCha20 method.
- (§2.6) transforming a 64-bit protocol nonce into 96 bits — protocol-level;
  QUIC always supplies 96-bit nonces.
- (§2.7) a PRF for ChaCha/Poly1305 crypto suites — the spec defines none.
- (§2.8) the P_MAX/A_MAX message-size limits — QUIC AEAD inputs are bounded
  by the MTU, far below the 256 GB / 2^64-1 limits.
- (§3) performance advice (state-copy reuse, big-number library choice) —
  informational.
- (§4) nonce-generation strategies (counters, LFSRs, encrypted counters) —
  operator/protocol guidance; nonce uniqueness itself is 8439-006.
