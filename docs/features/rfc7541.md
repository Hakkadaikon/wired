# RFC 7541 — HPACK: Header Compression for HTTP/2

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc7541.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This SDK speaks HTTP/3, so HPACK is not used as a whole: QPACK (RFC 9204)
reuses only HPACK's primitive types — the prefixed-integer representation
(§5.1) and the string literal / Huffman coding (§5.2, Appendix B). This
ledger covers that reused slice; the HPACK indexing machinery it replaces
is grouped under Out of scope.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 15/16 tested, 1 indirect, 0 untested.**

## §5.1 Integer Representation

- [x] 7541-001 (§5.1) When encoding an integer value strictly less than
  2^N-1, the implementation shall encode the value within the N-bit prefix.
  - test: `tests/app/qpack_test.c` — `test_qpack_integer_roundtrip`
- [x] 7541-002 (§5.1) When encoding an integer value of 2^N-1 or more, the
  implementation shall fill the prefix with ones and encode the remainder in
  7-bit groups, each octet carrying a continuation bit except the last.
  - test: `tests/app/qpack_test.c` — `test_qpack_integer_vector`
- [x] 7541-003 (§5.1) When decoding, the implementation shall recover the
  value by adding 2^N-1 to the concatenated 7-bit continuation groups
  (1337 with a 5-bit prefix decodes from 0x1F 0x9A 0x0A, Appendix C.1.2).
  - test: `tests/app/qpack_test.c` — `test_qpack_integer_vector`
  - test: `tests/app/qpack_test.c` — `test_qpack_integer_roundtrip`
- [x] 7541-004 (§5.1) The implementation shall support prefix sizes between
  1 and 8 bits, preserving the representation's pattern bits above the
  prefix.
  - test: `tests/app/literal_test.c` — `test_literal_name_prefix_boundary`
  - test: `tests/app/fieldline_test.c` —
    `test_fieldline_indexed_prefix_boundary`
- [x] 7541-005 (§5.1) If an integer encoding is cut before its final
  continuation octet, then the implementation shall treat it as a decoding
  error.
  - test: `tests/app/qpack_instruction_test.c` — `test_qpack_instr_truncated`
  - test: `tests/app/qpack_prefix_test.c` — `test_qpack_prefix_truncated`
- [~] 7541-006 (§5.1) If an integer encoding exceeds implementation limits
  in value or octet length, then the implementation shall treat it as a
  decoding error.
  - evidence: the decoder rejects continuation shifts past 56 bits
    (`take_group` in `src/app/qpack/qpack/integer.c`), bounding the value to
    64 bits; no dedicated over-long-encoding test.

## §5.2 String Literal Representation

- [x] 7541-007 (§5.2) The implementation shall decode a string literal's
  header as the H flag followed by the octet count as a 7-bit-prefix
  integer.
  - test: `tests/app/qpack_test.c` — `test_qpack_string`
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_string_h1`
- [x] 7541-008 (§5.2) When H is 0, the implementation shall take the string
  data as the raw octets of the string literal.
  - test: `tests/app/qpack_test.c` — `test_qpack_string`
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_string_h1`
- [x] 7541-009 (§5.2) When H is 1, the implementation shall Huffman-decode
  the string data using the code defined in Appendix B.
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_string_h1`
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_litname_hname`
- [x] 7541-010 (§5.2) If the string data is shorter than the declared
  length, then the implementation shall treat the literal as malformed.
  - test: `tests/app/qpack_test.c` — `test_qpack_string`
- [x] 7541-011 (§5.2) When encoding a string literal, the implementation
  shall emit an H flag and length that match the encoded data (this SDK
  emits raw octets, H=0).
  - test: `tests/app/qpack_test.c` — `test_qpack_string`
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_string_h1`
- [x] 7541-012 (§5.2) Upon decoding, the implementation shall discard an
  incomplete code of up to 7 bits at the end of the encoded data as padding.
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_padding`
- [x] 7541-013 (§5.2) If the padding does not correspond to the most
  significant bits of the EOS code (not all ones), then the implementation
  shall treat it as a decoding error.
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_bad_padding`
- [x] 7541-014 (§5.2) If the padding is strictly longer than 7 bits, or the
  encoded data contains the EOS symbol itself, then the implementation
  shall treat it as a decoding error.
  - test: `tests/app/qpack_huffman_test.c` — `test_huffman_eos_rejected`

## Out of scope

HPACK machinery that QPACK (RFC 9204) replaces with its own equivalents,
plus HTTP/2-only and encoder-optional behavior, excluded from the coverage
denominator:

- (§2.1–§2.4, §3) Header block processing, ordering, and the combined index
  address space — QPACK defines its own field-section processing and
  indexing.
- (§4, §6.3) Dynamic table management, eviction, and the dynamic table size
  update instruction — QPACK ships its own dynamic table and instructions.
- (§6.1–§6.2) HPACK header field representations (indexed, literal with /
  without / never indexed) — replaced by QPACK field-line representations.
- (Appendix A) The HPACK static table — QPACK defines its own static table.
- (§5.2) Huffman *encoding* of emitted strings — an encoder's option; this
  SDK always emits raw (H=0) string literals, which the format permits.
- (§7.1) Dynamic-table probing mitigations and never-indexed-literal
  forwarding policy — the HPACK dynamic table is not shipped; forwarding is
  an intermediary concern.
- (§7.2–§7.3) Static-Huffman information-leak discussion and
  SETTINGS_HEADER_TABLE_SIZE memory tuning — informational / HTTP/2
  operator concerns.
