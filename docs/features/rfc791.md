# RFC 791 — Internet Protocol (IPv4)

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc791.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Scope: this ledger covers only the IPv4 header build/parse and checksum
requirements exercised by wired's own AF_XDP path
(`src/transport/io/socket/net/ipv4.c`). wired is a server SDK, an endpoint,
not a gateway: routing, fragmentation-by-gateways, and options are out of
scope and listed below rather than silently dropped. The header is built by
wired itself only on the AF_XDP path; the interop runs used kernel UDP
sockets, which build their own IPv4 headers, so this framing is
unit-tested but not interop-proven.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 5/10 tested, 2 indirect, 3 untested.**

## §3.1 Internet Header Format

- [x] 791-001 (§3.1) The implementation shall set the Version field to 4.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`
- [ ] 791-002 (§3.1) The implementation shall set the Internet Header
  Length field to the header length in 32-bit words, with a minimum
  correct value of 5.
- [~] 791-003 (§3.1) The implementation shall set Total Length to the
  length of the datagram in octets, including the internet header and
  data.
  - evidence: `quic_ipv4_build` writes the caller-supplied `total_len` field
    verbatim (`src/transport/io/socket/net/ipv4.c`) and
    `test_net_datagram_over_link` (`tests/transport/net_test.c`) constructs
    it as header-plus-UDP-length; no test asserts rejection of a mismatched
    Total Length on parse.
- [ ] 791-004 (§3.1) The implementation shall set the Time to Live field to
  an upper bound, in seconds, on the datagram's lifetime, decremented at
  each point of processing.
- [x] 791-005 (§3.1) The implementation shall set the Protocol field to the
  next-level protocol carried in the data portion of the datagram.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`
- [x] 791-006 (§3.1) The implementation shall compute the Header Checksum
  as the 16-bit one's complement of the one's complement sum of all 16-bit
  words in the header, with the checksum field itself treated as zero
  during the computation.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`
  - test: `tests/transport/net_test.c` — `test_net_datagram_over_link`
- [x] 791-007 (§3.1) The implementation shall recompute and verify the
  Header Checksum at each point that the internet header is processed.
  - test: `tests/transport/net_test.c` — `test_net_datagram_over_link`
- [x] 791-008 (§3.1) The implementation shall set the Source Address and
  Destination Address fields to the 32-bit source and destination internet
  addresses.
  - test: `tests/transport/net_test.c` — `test_ipv4_checksum`

## §3.2 Discussion

- [ ] 791-009 (§3.2) The implementation shall be conservative in the
  datagrams it sends (well-formed) and liberal in the datagrams it accepts
  (not rejecting technical errors where the meaning is still clear).
- [~] 791-010 (Introduction) If the Header Checksum fails, then the
  implementation shall discard the datagram at once.
  - evidence: `quic_ipv4_check` returns false on a corrupted header
    (`test_ipv4_checksum` flips a source-address byte and observes the
    check fail, `tests/transport/net_test.c`); no test demonstrates the
    caller actually discarding the datagram on that false result.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator, per this document's
scope note above:

- (§1.4, §2.3, §3.1 Options) The Options mechanism (security, loose/strict
  source routing, record route, timestamp) and the Padding field — control
  functions for gateways and special routing scenarios; wired's AF_XDP path
  builds only the fixed 20-byte header with no options.
- (§1.4, §2.3, §3.1 Flags/Fragment Offset, §3.2 Fragmentation and
  Reassembly) Fragmentation and reassembly of datagrams (the DF/MF flags,
  Fragment Offset field, and the fragmentation/reassembly procedures) —
  gateway and small-packet-network behavior; explicitly out of scope for
  this endpoint SDK per the task scoping note, and QUIC's own design avoids
  depending on IP fragmentation.
- (§1.4, §2.3, §2.4) Routing and gateway forwarding of datagrams between
  networks, and the Gateway to Gateway Protocol — router/gateway behavior,
  not an endpoint concern.
- (§3.1 Type of Service) The Type of Service field's precedence, delay,
  throughput, and reliability indications — a hint to intermediate network
  selection that wired's endpoint role does not act on; the field is zeroed
  by `quic_ipv4_build`.
- (§3.1 Identification) The Identification field's use in distinguishing
  fragments of one datagram from another — meaningful only in conjunction
  with the out-of-scope fragmentation mechanism above.
- (§3.3 Interfaces) The example SEND/RECV upper-level interface between IP
  and its user — a descriptive API shape for illustration, not a wire
  requirement.
- (Appendix A, Appendix B) Worked examples and data transmission bit order
  — illustrative and already reflected in the RFC 1071 ledger's checksum
  worked example.
- (Glossary, References) Definitional and bibliographic material —
  non-normative.
