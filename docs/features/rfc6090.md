# RFC 6090 ‚Äî Fundamental Elliptic Curve Cryptography Algorithms

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc6090.txt`, not in git), scoped to the P-256 field and
group arithmetic the SDK ships (`src/crypto/asymmetric/ecc/p256/`) and its
ECDSA (KT-I) verification use. Each requirement carries the test that
demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` ‚Äî demonstrated by the referenced test
- `[~]` ‚Äî exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` ‚Äî not demonstrated by any test yet

**Coverage: 18/22 tested, 2 indirect, 2 untested.**

## ¬ß3 Elliptic Curve Groups

- [x] 6090-005 (¬ß3) The implementation shall treat a pair (x, y) in Fp as a
  point only when it satisfies the curve equation y^2 = x^3 + a*x + b.
  - test: `tests/crypto/p256_point_test.c` ‚Äî `test_p256_g_on_curve`
- [~] 6090-006 (¬ß3) If a coordinate pair does not satisfy the curve
  equation, then the implementation shall reject it as a public key.
  - evidence: `quic_ecdsa_p256_verify` validates the point with
    `quic_ec_on_curve` before use; no test feeds an off-curve point.
- [x] 6090-007 (¬ß3) When one operand of the group operation is the point at
  infinity, the implementation shall return the other operand.
  - test: `tests/crypto/p256_point_test.c` ‚Äî `test_p256_add_infinity_identity`
  - test: `tests/crypto/p256_point_test.c` ‚Äî `test_p256_add_infinity_both`
- [~] 6090-008 (¬ß3) When P is not equal to Q and x1 equals x2 (inverse
  points), the implementation shall return the point at infinity.
  - evidence: `test_p256_scalar`'s n*G = infinity result exercises the
    Jacobian same-x/opposite-y branch; the affine branch has no dedicated
    test.
- [x] 6090-009 (¬ß3) When P is not equal to Q and x1 differs from x2, the
  implementation shall compute x3 = ((y2-y1)/(x2-x1))^2 - x1 - x2 and
  y3 = (x1-x3)*(y2-y1)/(x2-x1) - y1.
  - test: `tests/crypto/ecdsa_verify_test.c` ‚Äî `test_ecdsa_valid`
- [x] 6090-010 (¬ß3) When P equals Q and y1 is nonzero, the implementation
  shall compute x3 = ((3*x1^2 + a)/(2*y1))^2 - 2*x1 and
  y3 = (x1-x3)*(3*x1^2 + a)/(2*y1) - y1.
  - test: `tests/crypto/p256_point_test.c` ‚Äî `test_p256_double_eq_add`
- [x] 6090-011 (¬ß2.2/¬ß3) The implementation shall compute scalar multiples
  k*G by repeated doubling and addition, satisfying 1*G = G and
  n*G = the point at infinity.
  - test: `tests/crypto/p256_point_test.c` ‚Äî `test_p256_scalar`

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (¬ß3) Doubling a point with y1 = 0 ‚Äî unreachable on P-256: the
  prime-order group contains no point of order two.
- (¬ß3.1) Homogeneous-coordinate group-operation formulas ‚Äî the SDK uses
  Jacobian coordinates (¬ß3.2) instead.
- (¬ß3.3.1) Discriminant check ‚Äî parameter generation; the SDK hard-codes
  the fixed NIST P-256 parameters.
- (¬ß3.3.2) Parameter-choice guidance (weak-curve SHOULD NOTs) ‚Äî fixed
  P-256 parameter set, no parameter negotiation.
- (¬ß4) ECDH, including compact representation/output MUSTs ‚Äî the SDK's
  TLS key exchange uses X25519 (RFC 7748), not P-256 ECDH.
- (¬ß5.3, ¬ß5.5) KT-IV signatures and KT-IV-to-KT-I conversion ‚Äî not
  shipped; only the ECDSA (KT-I) form is used.
- (¬ß5.4.2) Signature creation with a randomly chosen k ‚Äî the SDK derives
  k deterministically (RFC 6979; see rfc6979.md).
- (¬ß7) Interoperability mappings to IKE and IEEE P1363 ‚Äî protocol
  profiles outside this SDK.
- (¬ß8.1) ECDH validation ‚Äî no P-256 ECDH to validate.
- (¬ß10) Security-considerations cryptanalysis background ‚Äî informative.
- (Appendix B) Random integer generation ‚Äî replaced by deterministic
  RFC 6979 nonce generation.
- (Appendices C‚ÄìF) Compact-representation proof, example parameter set,
  notation, and pseudocode ‚Äî informative.
