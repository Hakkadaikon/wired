# RFC 6090 — Fundamental Elliptic Curve Cryptography Algorithms

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc6090.txt`, not in git), scoped to the P-256 field and
group arithmetic the SDK ships (`src/crypto/asymmetric/ecc/p256/`) and its
ECDSA (KT-I) verification use. Each requirement carries the test that
demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 18/22 tested, 2 indirect, 2 untested.**

## §2.1/§2.3 Modular Arithmetic and the Finite Field Fp

- [x] 6090-001 (§2.1) The implementation shall compute a + b mod q as a + b
  when the sum is below q, and a + b - q otherwise.
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_addsub`
- [x] 6090-002 (§2.1) The implementation shall compute a - b mod q as a - b
  when the difference is non-negative, and a - b + q otherwise.
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_addsub`
- [x] 6090-003 (§2.1) The implementation shall compute a * b mod q as the
  remainder of a * b divided by q.
  - test: `tests/crypto/p256_field_test.c` —
    `test_p256_field_fast_matches_generic`
- [x] 6090-004 (§2.1) The implementation shall compute the modular inverse
  1/x mod q for every x coprime with q, for both the field prime p and the
  group order n.
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_inv`
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_mont_inv_n`

## §3 Elliptic Curve Groups

- [x] 6090-005 (§3) The implementation shall treat a pair (x, y) in Fp as a
  point only when it satisfies the curve equation y^2 = x^3 + a*x + b.
  - test: `tests/crypto/p256_point_test.c` — `test_p256_g_on_curve`
- [~] 6090-006 (§3) If a coordinate pair does not satisfy the curve
  equation, then the implementation shall reject it as a public key.
  - evidence: `quic_ecdsa_p256_verify` validates the point with
    `quic_ec_on_curve` before use; no test feeds an off-curve point.
- [ ] 6090-007 (§3) When one operand of the group operation is the point at
  infinity, the implementation shall return the other operand.
  - gap: `quic_ec_add` handles the identity operand (`add_infinity`), but
    no test adds the point at infinity explicitly.
- [~] 6090-008 (§3) When P is not equal to Q and x1 equals x2 (inverse
  points), the implementation shall return the point at infinity.
  - evidence: `test_p256_scalar`'s n*G = infinity result exercises the
    Jacobian same-x/opposite-y branch; the affine branch has no dedicated
    test.
- [x] 6090-009 (§3) When P is not equal to Q and x1 differs from x2, the
  implementation shall compute x3 = ((y2-y1)/(x2-x1))^2 - x1 - x2 and
  y3 = (x1-x3)*(y2-y1)/(x2-x1) - y1.
  - test: `tests/crypto/ecdsa_verify_test.c` — `test_ecdsa_valid`
- [x] 6090-010 (§3) When P equals Q and y1 is nonzero, the implementation
  shall compute x3 = ((3*x1^2 + a)/(2*y1))^2 - 2*x1 and
  y3 = (x1-x3)*(3*x1^2 + a)/(2*y1) - y1.
  - test: `tests/crypto/p256_point_test.c` — `test_p256_double_eq_add`
- [x] 6090-011 (§2.2/§3) The implementation shall compute scalar multiples
  k*G by repeated doubling and addition, satisfying 1*G = G and
  n*G = the point at infinity.
  - test: `tests/crypto/p256_point_test.c` — `test_p256_scalar`

## §5.4.3 KT-I (ECDSA) Signature Verification

- [x] 6090-014 (§5.4.3) If a signature component s1 or s2 is zero, then the

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§3) Doubling a point with y1 = 0 — unreachable on P-256: the
  prime-order group contains no point of order two.
- (§3.1) Homogeneous-coordinate group-operation formulas — the SDK uses
  Jacobian coordinates (§3.2) instead.
- (§3.3.1) Discriminant check — parameter generation; the SDK hard-codes
  the fixed NIST P-256 parameters.
- (§3.3.2) Parameter-choice guidance (weak-curve SHOULD NOTs) — fixed
  P-256 parameter set, no parameter negotiation.
- (§4) ECDH, including compact representation/output MUSTs — the SDK's
  TLS key exchange uses X25519 (RFC 7748), not P-256 ECDH.
- (§5.3, §5.5) KT-IV signatures and KT-IV-to-KT-I conversion — not
  shipped; only the ECDSA (KT-I) form is used.
- (§5.4.2) Signature creation with a randomly chosen k — the SDK derives
  k deterministically (RFC 6979; see rfc6979.md).
- (§7) Interoperability mappings to IKE and IEEE P1363 — protocol
  profiles outside this SDK.
- (§8.1) ECDH validation — no P-256 ECDH to validate.
- (§10) Security-considerations cryptanalysis background — informative.
- (Appendix B) Random integer generation — replaced by deterministic
  RFC 6979 nonce generation.
- (Appendices C–F) Compact-representation proof, example parameter set,
  notation, and pseudocode — informative.
