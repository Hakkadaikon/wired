# RFC 6125 — Representation and Verification of Service Identity

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc6125.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger is scoped to the SAN/hostname matching API this server SDK
ships (`quic_x509_san_matches`, RFC 5280 §4.2.1.6 dNSName entries only) —
the API is caller-invoked (see [Security](../security.md)); certificate
issuance guidance (§4, aimed at CAs) and reference-identifier construction
from application inputs (§6.1–§6.3, a caller-side, non-DNS-ID identifier
policy) are covered only where the shipped matcher's behavior bears on
them. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 6/11 tested, 1 indirect, 4 untested.**

## §4.1 Representing Server Identity (rules bearing on the shipped matcher)

- [~] 6125-001 (§4.1 rule 7) Unless continued support for the wildcard
  character is allowed, the DNS domain name portion of a presented
  identifier should not contain '*' other than as the complete left-most
  label.
  - evidence: `quic_x509_san_matches` (via `is_wildcard`/`wildcard_match`
    in `src/crypto/pki/encoding/x509/san.c`) only special-cases a SAN
    entry whose first two bytes are exactly `"*."`; a `*` anywhere else
    in the entry is matched literally (as an ordinary character) rather
    than as a wildcard, so fragment-wildcard entries (`fo*.example.com`)
    never match — `tests/crypto/san_test.c` covers the accepted `*.`
    shape only, not a literal-`*`-mismatch case.

## §6.3 Preparing to Seek a Match

- [x] 6125-002 (§6.3) The search succeeds if any presented identifier
  matches one of the reference identifiers.
  - test: `tests/crypto/san_test.c` — `test_exact_match`
- [x] 6125-003 (§6.3) The search fails if the reference identifier list is
  exhausted without finding a match.
  - test: `tests/crypto/san_test.c` — `test_no_match`
  - test: `tests/crypto/san_test.c` — `test_no_san`

## §6.4.1 Checking of Traditional Domain Names

- [x] 6125-004 (§6.4.1) The implementation shall match the DNS domain name
  portion by comparing domain name labels using case-insensitive ASCII
  comparison.
  - test: `tests/crypto/san_test.c` — `test_san_case_fold`
- [x] 6125-005 (§6.4.1) Each label must match in order for the names to be
  considered to match (subject to the wildcard rule of §6.4.3).
  - test: `tests/crypto/san_test.c` — `test_no_match`

## §6.4.2 Checking of Internationalized Domain Names

- [ ] 6125-006 (§6.4.2) If the DNS domain name portion is an
  internationalized domain name, the implementation shall convert any
  U-labels to A-labels before checking, and compare A-labels as
  case-insensitive ASCII.
  - gap: `quic_x509_san_matches` treats hostnames as opaque ASCII bytes;
    no U-label-to-A-label (Punycode) conversion exists in `src/`.

## §6.4.3 Checking of Wildcard Certificates

- [x] 6125-007 (§6.4.3 rule 1) The implementation should not match a
  presented identifier in which the wildcard character comprises a label
  other than the left-most label.
  - test: `tests/crypto/san_test.c` — `test_wildcard_no_nested`
  - evidence: `is_wildcard` only recognizes `*` as the first two bytes of
    the whole entry (the left-most label position); a `*` in a
    non-left-most label is never treated as a wildcard.
- [x] 6125-008 (§6.4.3 rule 2) If the wildcard character is the only
  character of the left-most label in the presented identifier, the
  implementation should compare against only the left-most label of the
  reference identifier.
  - test: `tests/crypto/san_test.c` — `test_wildcard_match`
  - test: `tests/crypto/san_test.c` — `test_wildcard_no_nested`
- [ ] 6125-009 (§6.4.3 rule 3) The implementation may match a presented
  identifier in which the wildcard character is not the only character of
  the label (e.g. `baz*.example.net`).
  - gap: `wildcard_match` requires the left-most label to be exactly `*`
    (`is_wildcard` checks `entry.p[0]=='*' && entry.p[1]=='.'`); a
    fragment wildcard is not recognized as a wildcard at all (see
    6125-001) and is instead compared byte-for-byte, so this optional
    rule is deliberately not implemented, and no test exercises it.

## §6.4.4 Checking of Common Names

- [ ] 6125-010 (§6.4.4) A client must not seek a match for a CN-ID
  reference identifier if the presented identifiers include a DNS-ID,
  SRV-ID, URI-ID, or any application-specific identifier type the client
  supports.
  - gap: `quic_x509_san_matches` only reads subjectAltName dNSName
    entries; it never falls back to (or even reads) the Common Name, so
    the ordering constraint has no corresponding code path to test.
- [ ] 6125-011 (§6.4.4) If and only if the presented identifiers include
  no DNS-ID/SRV-ID/URI-ID/application-specific identifier, the client may
  as a last resort check a CN-ID, applying the DNS-domain-name comparison
  rules of §6.4.1–§6.4.3.
  - gap: no CN-ID fallback path exists in `src/` (see 6125-010); a
    certificate with no SAN extension is always rejected —
    `tests/crypto/san_test.c test_no_san` demonstrates the "no CN-ID
    fallback" behavior as a rejection, not a supported CN-ID check.

## Out of scope

Caller-side identifier-construction policy, non-DNS identifier types, and
CA-facing issuance guidance this server SDK does not implement as code
(the matching API operates only on an already-decided hostname against a
certificate's dNSName SANs), excluded from the coverage denominator:

- (§4) Representing Server Identity (CA/issuer-facing rules for what
  identifiers a certificate SHOULD contain) — guidance for certificate
  issuers, not verifier code this SDK ships; §4.1 rule 7 is covered above
  only because the shipped matcher's wildcard handling bears on it.
- (§5) Requesting Server Certificates — CSR-construction guidance for
  service operators, not SDK code.
- (§6.1) Overview / high-level client verification steps — descriptive
  framing, restated by the numbered subsections below it.
- (§6.2) Constructing a List of Reference Identifiers — this is a
  caller/application policy decision (which hostname(s) to check against,
  derived from a URI, account configuration, or user input); the SDK's
  `quic_x509_san_matches` takes the already-decided hostname as an
  argument and does not construct reference identifiers itself.
- (§6.3, CN-ID / pinning aspects) The "MUST NOT construct a reference
  identifier for non-CN RDNs" and certificate-pinning fallback (§6.6.2,
  §6.6.4) — pinning and reference-identifier policy are caller
  responsibilities; the matcher exposes only a single-hostname match
  primitive.
- (§6.5, §6.5.1, §6.5.2) SRV-ID and URI-ID matching (application service
  type portion) — the SDK matches only DNS-ID-style dNSName SAN entries;
  SRV-ID and URI-ID are not extracted or compared.
- (§6.6) Outcome (match found / pinned certificate / no match / fallback)
  — the overall accept/reject decision and human-facing fallback behavior
  belong to the caller that invokes `quic_x509_san_matches`, per
  `docs/security.md`'s "Caller responsibility" note.
- (§7.1) Pinned Certificates — pinning policy is a caller/application
  concern.
- (§7.3) Internationalized Domain Names (security discussion) —
  inapplicable while IDN conversion itself is not implemented (6125-006).
- (§7.4) Multiple Identifiers — discussion of certificate-issuance
  practice (embedding multiple DNS-IDs vs. relying on SNI), not verifier
  code.
