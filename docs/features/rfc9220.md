# RFC 9220 — Bootstrapping WebSockets with HTTP/3

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9220.txt`, not in git). RFC 9220 is a short document that
adapts RFC 8441's Extended CONNECT mechanism (defined for HTTP/2) to
HTTP/3; this SDK implements the Extended CONNECT mechanism itself (the
`:protocol` pseudo-header and the SETTINGS_ENABLE_CONNECT_PROTOCOL
setting), which WebTransport (see
[draft-ietf-webtrans-http3](draft-webtrans-http3.md)) builds on. Each
requirement carries the test that demonstrates it; an unchecked box with no
test line is an open gap. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 9/11 tested, 1 indirect, 1 untested.**

## §3 WebSockets Upgrade over HTTP/3

  CONNECT_PROTOCOL identifier as 0x08, the same value as HTTP/2.

    `test_qpack_settings_connect_protocol_id`

  - test: `tests/app/connect_test.c` — `test_connect_protocol_negotiation`

- [ ] 9220-006 (§3) If a server advertises support for Extended CONNECT
  but receives an Extended CONNECT request with a `:protocol` value that
  is unknown or unsupported, then the server should respond with a 501

    `test_srvrun_wt_connect_missing_authority_no_session`

  - test: `tests/app/srvrun_test.c` —

  - test: `tests/app/srvrun_test.c` —

- [x] 9220-009 (§3) The HTTP/3 stream closure of an Extended CONNECT
  stream shall be analogous to a TCP connection closure: an orderly close

  - test: `tests/app/srvrun_test.c` —

- [~] 9220-010 (§3) A `:protocol` value the server does not recognize
  shall not be accepted for a WebTransport session establishment.
  - evidence: `srvrun_is_wt_connect` (srvrun.c) only recognizes the
    `webtransport`/`webtransport-h3` tokens (`wt_protocol_is_webtransport`);

    ok`'s general negotiation check, exercised by
    `tests/app/connect_test.c`'s `test_connect_protocol_negotiation`, but no
    test feeds an unrecognized non-empty `:protocol` value specifically

- [x] 9220-011 (§3) The `:protocol` pseudo-header field shall encode as a

  no entry for `:protocol`.
  - test: `tests/app/pseudo_encode_test.c` —
    `test_pseudo_protocol_roundtrip`

## Out of scope

Requirements this document defers entirely to RFC 8441 (referenced, not
reproduced, by RFC 9220 itself) or that are client-only:

- (§3) The full semantics of the `:protocol` pseudo-header field and the
  `:path`/`:authority` pseudo-header field reinterpretation under Extended
  CONNECT, beyond what RFC 9220's own text states, are defined in RFC 8441
  (HTTP/2) and adopted "identical" by reference; RFC 8441 is not one of
  this ledger's three source specs and is not separately re-extracted here.
- (§3) Problem-details responses (RFC 7807) accompanying a 501 — optional
  (MAY) elaboration on an already-unimplemented 501 path.
- (§3) Client-side interpretation of the SETTINGS_ENABLE_CONNECT_PROTOCOL
  setting and client-side Extended CONNECT request construction — client
  behavior.
