# RFC 8017 — PKCS #1: RSA Cryptography Specifications v2.2

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8017.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger is scoped to what this SDK implements: RSASSA-PKCS1-v1_5
signature verification (§8.2/§9.2) and RSASSA-PSS/MGF1 signature
verification (§8.1/§9.1/Appendix B.2.1), both used only to verify a
certificate chain's or CertificateVerify's RSA signature. Encryption
schemes (§7, RSAES-*), key generation, and RSA signature *generation*
(RSASP1, §8.1.1/§8.2.1) are absent from `src/crypto/` (grep-verified) and
are grouped under Out of scope. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 19/22 tested, 3 indirect, 0 untested.**

## §3.1 RSA Public Key

- [x] 8017-001 (§3.1) The implementation shall represent an RSA public key
  as the pair (n, e), the modulus and the public exponent, both positive
  integers.
  - test: `tests/crypto/rsa_pubkey_test.c` — `test_rsa_pubkey_extract`
- [x] 8017-002 (§3.1) The RSA public exponent e shall be an integer
  between 3 and n - 1.
  - test: `tests/crypto/rsa_pubkey_test.c` — `test_rsa_pubkey_e_min`
  - test: `tests/crypto/rsa_pubkey_test.c` — `test_rsa_pubkey_e_max`
- [~] 8017-003 (§3.1) The RSA modulus n shall be a product of distinct odd
  primes.
  - evidence: `quic_x509_rsa_pubkey` rejects an even modulus
    (`tests/crypto/rsa_pubkey_test.c test_rsa_pubkey_even_n`) as a
    necessary condition; primality/distinctness of the factors is not
    (and cannot be, from the public key alone) verified — no dedicated
    factorization test, matching §6's explicit note that key validation
    is outside an implementation's scheme operations.

## §4 Data Conversion Primitives (I2OSP / OS2IP)

- [x] 8017-004 (§4.1, §4.2) The implementation shall convert between a
  nonnegative integer and a fixed-length big-endian octet string
  (I2OSP/OS2IP), the most significant octet first.
  - test: `tests/crypto/bignum_test.c` — `test_bn_be_roundtrip`
  - evidence: `quic_bn_from_be`/`quic_bn_to_be` (`src/crypto/asymmetric/
    bignum/bignum.h`) implement OS2IP/I2OSP and are exercised end-to-end
    by every RSA verify test (signature and modulus are read via
    `quic_bn_from_be`, the recovered message via `quic_bn_to_be`).

## §9.1 EMSA-PSS

- [x] 8017-010 (§9.1.2 step 3) If emLen < hLen + sLen + 2, the
  implementation shall output "inconsistent".
  - test: `tests/crypto/rsa_pss_test.c` — `test_rsa_pss_sizes`
- [~] 8017-011 (§9.1.2 step 4) If the rightmost octet of EM is not 0xbc,
  the implementation shall output "inconsistent".
  - evidence: `pss_prefix_bad` in `src/crypto/asymmetric/rsa/pss.c` checks
    `em.p[em.n - 1] != 0xbc` before any other step; exercised implicitly
    by every valid-signature test (a genuine PSS EM always ends 0xbc) and
    by `test_rsa_pss_tampered_sig`/`test_rsa_pss_wrong_hash`, which
    corrupt other fields and still route through this same gate — no
    test flips the trailer byte specifically.
- [x] 8017-012 (§9.1.2 steps 5-9) The implementation shall split EM into
  maskedDB and H, generate dbMask = MGF(H, emLen-hLen-1), unmask
  DB = maskedDB XOR dbMask, and clear the leftmost 8emLen-emBits bits of
  DB.
  - test: `tests/crypto/rsa_pss_test.c` — `test_rsa_pss_valid`
  - test: `tests/crypto/mgf1_test.c` — `test_mgf1_kat`
- [~] 8017-013 (§9.1.2 step 6) If the leftmost 8emLen-emBits bits of the
  leftmost octet of maskedDB are not all zero, the implementation shall
  output "inconsistent".
  - evidence: `pss_top_bits_clear` (`src/crypto/asymmetric/rsa/pss.c`)
    implements this exact mask-and-compare; reachable only via a
    corrupted EM, which no test constructs directly — covered
    transitively by every EMSA-PSS decode but not targeted.
- [x] 8017-014 (§9.1.2 step 10) If the leftmost emLen-hLen-sLen-2 octets of
  DB are not zero, or the following octet is not 0x01, the implementation
  shall output "inconsistent".
  - test: `tests/crypto/rsa_pss_test.c` — `test_rsa_pss_tampered_sig`
  - evidence: `pss_padding_ok` implements the PS/0x01 check; the tampered-
    signature test flips a signature byte, which (through the RSA
    exponentiation) perturbs the recovered EM and is caught by this or
    the H'=H check, without isolating which gate fires.
- [x] 8017-015 (§9.1.2 steps 11-14) The implementation shall recompute
  H' = Hash(0x00*8 || mHash || salt) and output "consistent" iff H' = H.
  - test: `tests/crypto/rsa_pss_test.c` — `test_rsa_pss_valid`
  - test: `tests/crypto/rsa_pss_test.c` — `test_rsa_pss_wrong_hash`

## §9.2 EMSA-PKCS1-v1_5

- [x] 8017-020 (§9.2 step 2) The implementation shall embed the digest
  algorithm identifier via the DigestInfo DER prefix (SEQUENCE {
  digestAlgorithm, digest }) matching the message's hash function.
  - test: `tests/crypto/rsa_verify_test.c` — `test_rsa_pkcs1_valid`
  - evidence: `rsa_di_sha256`/`rsa_di_sha384`/`rsa_di_sha512` in
    `src/crypto/asymmetric/rsa/rsa_verify.c` reproduce the RFC 8017 §9.2
    Note 1 DigestInfo prefixes byte-for-byte for the three hash lengths
    this SDK supports.
- [x] 8017-021 (§9.2 step 3) If emLen < tLen + 11, the implementation
  shall reject the encoding as "RSA modulus too short".
  - test: `tests/crypto/rsa_verify_test.c` — `test_rsa_pkcs1_sizes`
  - evidence: `n_len_bad` (`n_len < 11 + RSA_DI_LEN + hash_len`)
    implements this floor.
- [x] 8017-022 (§9.2 steps 4-5) The implementation shall build
  EM = 0x00 || 0x01 || PS || 0x00 || T, where PS is emLen-tLen-3 octets of
  0xff (at least 8 octets).
  - test: `tests/crypto/rsa_verify_test.c` — `test_rsa_pkcs1_valid`
  - evidence: `emsa_pkcs1` in `src/crypto/asymmetric/rsa/rsa_verify.c`
    constructs exactly this layout and enforces the 8-octet PS minimum
    (`if (em_len < fixed + 8) return 0`).

## Out of scope

Encryption schemes, key generation, and signature *generation*, none of
which `src/crypto/` implements (grep-verified absent), excluded from the
coverage denominator:

- (§3.2) RSA Private Key representation (both forms) and the CRT
  coefficients — no private-key type exists in `src/`; this SDK only
  verifies signatures with a public key.
- (§5.1) RSAEP / RSADP (Encryption/Decryption Primitives) — no RSA
  encryption or decryption is shipped.
- (§5.2.1) RSASP1 (Signature Primitive) — no RSA signature *generation*
  exists; the SDK's own certificates and CertificateVerify use ECDSA/
  Ed25519 (see [Security](../security.md)), and RSA is verify-only for
  peer/CA-issued material.
- (§6) Overview of Schemes' general encryption-scheme framing, and the
  "do not reuse a key pair across schemes" recommendation for
  signature+encryption combinations — inapplicable without an encryption
  scheme.
- (§7, §7.1, §7.2) Encryption Schemes: RSAES-OAEP and RSAES-PKCS1-v1_5 (and
  their encoding methods EME-OAEP/EME-PKCS1-v1_5) — not shipped.
- (§10) Security Considerations — a pointer to considerations discussed
  throughout the document; not independently actionable.
- (Appendix A.1.2, A.2.*) ASN.1 syntax for RSA private keys and PKCS #1
  algorithm identifiers beyond RSAPublicKey (A.1.1, covered via
  `quic_x509_rsa_pubkey`) — no private-key ASN.1 parsing is shipped.
- (Appendix B.1) Hash function OIDs and DER prefixes for MD2, MD5, SHA-1,
  SHA-224, SHA-512/224, SHA-512/256 — this SDK's DigestInfo table
  (`rsa_di[]`) supports only SHA-256/384/512, matching the hash functions
  the rest of the SDK implements (FIPS 180-4); the legacy/unsupported
  hashes are rejected by `di_prefix` returning 0, not implemented as
  additional encodings.
- (Appendix B.2.2) MGF2 (multi-hash mask generation function) — only
  MGF1-SHA-256 is shipped, matching TLS 1.3's rsa_pss_rsae_sha256.
