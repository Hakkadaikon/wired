# RFC 8410 — Ed25519/X25519 Algorithm Identifiers for X.509

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8410.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Scope: what this TLS server SDK's certificate stack consumes/produces —
Ed25519 SubjectPublicKeyInfo encoding, EdDSA signature encoding, and
self-signed certificate construction. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 7/16 tested, 1 indirect, 8 untested.**

## §3 Curve25519 and Curve448 Algorithm Identifiers

- [ ] 8410-001 (§3) The implementation shall recognize the id-X25519 OBJECT
  IDENTIFIER (1.3.101.110) identifying the X25519 key-agreement algorithm.
- [ ] 8410-002 (§3) The implementation shall recognize the id-X448 OBJECT
  IDENTIFIER (1.3.101.111) identifying the X448 key-agreement algorithm.
- [x] 8410-003 (§3) The implementation shall recognize the id-Ed25519
  OBJECT IDENTIFIER (1.3.101.112) identifying the Ed25519 signature
  algorithm.
  - test: `tests/crypto/selfcert_test.c` — `test_tbs_spki_key`
  - test: `tests/crypto/selfcert_test.c` — `test_build_selfsigned`
- [ ] 8410-004 (§3) The implementation shall recognize the id-Ed448 OBJECT
  IDENTIFIER (1.3.101.113) identifying the Ed448 signature algorithm.
- [x] 8410-005 (§3) For the id-X25519, id-X448, id-Ed25519, and id-Ed448
  algorithm identifiers, the implementation shall encode the
  AlgorithmIdentifier parameters as absent.
  - test: `tests/crypto/selfcert_test.c` — `test_tbs_spki_key`

## §4 Subject Public Key Fields

- [x] 8410-006 (§4) The implementation shall encode the Ed25519
  subjectPublicKey as the raw 32-byte public key octet string, with no
  additional ASN.1 wrapping, inside the SubjectPublicKeyInfo BIT STRING.
  - test: `tests/crypto/selfcert_test.c` — `test_tbs_spki_key`
- [x] 8410-007 (§4) The implementation shall encode the subjectPublicKey
  BIT STRING with a leading 0x00 unused-bits octet before the raw key
  bytes.
  - test: `tests/crypto/selfcert_test.c` — `test_tbs_spki_key`
  - test: `tests/crypto/selfcert_test.c` — `test_build_selfsigned`

## §5 Key Usage Bits

- [ ] 8410-008 (§5) Where the keyUsage extension is present on a
  certificate whose SubjectPublicKeyInfo indicates id-X25519 or id-X448,
  the implementation shall require keyAgreement.
- [ ] 8410-009 (§5) Where the keyUsage extension is present on an
  end-entity certificate whose SubjectPublicKeyInfo indicates id-Ed25519 or
  id-Ed448, the implementation shall require nonRepudiation and/or
  digitalSignature.
- [ ] 8410-010 (§5) Where the keyUsage extension is present on a CA
  certificate whose SubjectPublicKeyInfo indicates id-Ed25519 or id-Ed448,
  the implementation shall require one or more of nonRepudiation,
  digitalSignature, keyCertSign, and cRLSign.

## §6 EdDSA Signatures

- [x] 8410-011 (§6) The implementation shall use the same id-Ed25519
  algorithm identifier for the certificate's signatureAlgorithm as is used
  for the subject public key.
  - test: `tests/crypto/selfcert_test.c` — `test_tbs_spki_key`
  - test: `tests/crypto/selfcert_test.c` — `test_build_selfsigned`
- [~] 8410-012 (§6) When encoding an EdDSA signature, the implementation
  shall write the AlgorithmIdentifier's parameters as absent.
  - evidence: `quic_selfcert_tbs`'s `build_alg` emits
    SEQUENCE{ id-Ed25519 } as the tbs signature AlgorithmIdentifier (no
    second element), and `quic_selfcert_build` reuses the identical
    `build_alg` helper for the outer signatureAlgorithm — both exercised
    by `test_build_selfsigned`.
- [x] 8410-013 (§6) The implementation shall encode the EdDSA signature
  value ENC(R) || ENC(S) directly in the signatureValue BIT STRING, with
  no additional ASN.1 wrapping.
  - test: `tests/crypto/selfcert_test.c` — `test_build_selfsigned`

## §7 Private Key Format

- [ ] 8410-014 (§7) The implementation shall encode an Ed25519 private key
  as an OCTET STRING (CurvePrivateKey) directly holding the 32-byte seed,
  wrapped by the OneAsymmetricKey privateKey OCTET STRING field.
  - gap: the SDK's self-signed certificate path takes the 32-byte Ed25519
    seed as a caller-supplied buffer (`quic_selfcert_build(seed, ...)`
    /`quic_ed25519_keypair(seed, pub)`); no code encodes or decodes a
    OneAsymmetricKey/PKCS#8 DER structure for an Ed25519 private key
    (contrast with `wired_eckey_p256_priv`, which does parse SEC1/PKCS#8
    for P-256).

## Appendix A Invalid Encodings

- [x] 8410-015 (Appendix A) If the public key is encoded as an OCTET STRING
  rather than the required BIT STRING, then the implementation shall
  reject the SubjectPublicKeyInfo.
  - test: `tests/crypto/spki_test.c` — `test_spki_short_tbs`
  - evidence: `split_spki` requires the second SPKI element to tag-match
    `QUIC_DER_BIT_STRING`; a `quic_derseq_next_tagged` mismatch on any
    other tag (including OCTET STRING) fails the parse. The same codepath
    backs both P-256 and Ed25519 SPKI parsing (`quic_x509_public_key`),
    but no dedicated Ed25519-specific OCTET-STRING-substitution fixture
    exists.
- [ ] 8410-016 (Appendix A) The implementation shall enforce that an
  Ed25519/X25519 private-key masking/formatting requirement (e.g. the
  X25519 top/bottom bit-clamping rule) is applied when importing a private
  key.
  - gap: no private-key import/decode path exists for Ed25519 in this SDK
    (see 8410-014); the seed is used as supplied, and no masking/clamping
    check is performed or tested.

## Out of scope

Curves, algorithms, and key-management concerns this server SDK does not
implement, excluded from the coverage denominator:

- (§3, §4, §5, §9) X25519 and X448 key-agreement algorithm identifiers and
  their SubjectPublicKeyInfo/keyUsage encoding — the SDK has no ECDH-style
  key-agreement certificate path; `ed25519_field.c` implements curve25519
  field arithmetic shared with EdDSA signing only, not a standalone X25519
  KEX/SPKI feature.
- (§3, §4, §6, §9) Ed448 signature algorithm identifier and its
  SubjectPublicKeyInfo/signature encoding — the SDK implements only
  Ed25519; no Ed448/curve448 arithmetic exists in `src/crypto/`.
- (§7) OneAsymmetricKey/PKCS#8 encode+decode of Ed25519 private keys with
  the `attributes`/`publicKey` optional fields — the SDK consumes a raw
  32-byte seed directly; no ASN.1 private-key package is parsed or
  produced for Ed25519 (contrast P-256, which does have
  `wired_eckey_p256_priv` for SEC1/PKCS#8 import).
- (§8) Human-readable algorithm naming conventions ("ECDH", "X25519",
  "EdDSA", "Ed25519") — a display/logging convention for implementers, not
  a wire-format or parsing requirement.
- (§10) Illustrative worked examples (the sample public key, X25519
  certificate, and private-key dumps) — non-normative examples, not
  independently testable requirements beyond what is captured above.
- (§11) IANA module-identifier registration — a registry administration
  concern, not implementable code behavior.
- Appendix A's BER (vs. DER) decoding acceptance for OneAsymmetricKey — the
  SDK's ASN.1 layer (`crypto/pki/encoding/asn1/`) implements DER only, and
  RFC 8410 itself notes BER acceptance is optional ("some acceptors will
  only decode DER formats").
