# RFC 9221 — An Unreliable Datagram Extension to QUIC

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9221.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 22/27 tested, 2 indirect, 3 untested.**

## §3 Transport Parameter

- [x] 9221-001 (§3) Where datagram receive support is enabled, the server
  shall advertise it with the max_datagram_frame_size transport parameter

    `test_server_tp_datagram_frame_size`

  - test: `tests/tls/server_tp_test.c` —
    `test_server_tp_datagram_frame_size`

  - test: `tests/app/srvrun_test.c` —
    `test_srvrun_datagram_rejected_when_peer_unadvertised`

## §4 Datagram Frame Types

- [x] 9221-010 (§4) The implementation shall parse the DATAGRAM frame types
  0x30 and 0x31, taking the least significant bit of the Type field as the
  LEN bit that indicates whether a Length field is present.
  - test: `tests/app/datagram_test.c` — `test_datagram_with_len`
  - test: `tests/app/datagram_test.c` — `test_datagram_no_len`
  - test: `tests/transport/dispatch_state_test.c` — `test_dispatch_datagram`
- [x] 9221-011 (§4) When the LEN bit is 0, the implementation shall take
  the Datagram Data field as extending to the end of the packet.
  - test: `tests/app/datagram_test.c` — `test_datagram_no_len`
  - test: `tests/app/dg_recv_test.c` — `test_dg_recv_no_length`
- [x] 9221-012 (§4) When the LEN bit is 1, the implementation shall take
  the Length field as a variable-length integer giving the length of the
  Datagram Data field in bytes.
  - test: `tests/app/datagram_test.c` — `test_datagram_with_len`
  - test: `tests/app/dg_recv_test.c` — `test_dg_recv_with_length`
- [x] 9221-013 (§4) The implementation shall accept empty (zero-length)
  datagrams.
  - test: `tests/app/datagram_test.c` — `test_datagram_truncated`
  - test: `tests/app/dgcheck_test.c` — `test_dgcheck`
- [x] 9221-014 (§4) If a DATAGRAM frame's data is truncated relative to its
  declared Length, then the implementation shall reject the frame as
  malformed.
  - test: `tests/app/datagram_test.c` — `test_datagram_truncated`
  - test: `tests/app/dg_recv_test.c` — `test_dg_recv_truncated`
  - test: `tests/transport/dispatch_state_test.c` —
    `test_dispatch_datagram_malformed`

- [x] 9221-016 (§5) When a valid DATAGRAM frame is received, the server
  shall deliver the data to the application immediately.
  - test: `tests/app/srvrun_test.c` —
    `test_srvrun_rx_datagram_delivers_to_callback`
  - test: `tests/app/srvloop_test.c` —
    `test_srvloop_datagram_queued_on_step`
- [~] 9221-017 (§5, §6) The server shall protect DATAGRAM frames with
  either 0-RTT or 1-RTT keys.
  - evidence: the server extracts DATAGRAM frames only from decrypted 1-RTT
    payloads, and the wire round-trip test runs over sealed 1-RTT packets;
    no dedicated key-class assertion exists.
- [x] 9221-018 (§5) The implementation shall bound the datagram payload by
  the frame-size limit minus the type byte and, when present, the Length
  varint, since DATAGRAM frames cannot be fragmented.
  - test: `tests/app/dgsize_test.c` — `test_dgsize`

## Out of scope

Client-only, application-protocol, and optional-feature requirements this
server SDK does not implement, excluded from the coverage denominator:

- (§3) Storing the server's max_datagram_frame_size with 0-RTT state and
  validating that the new value is not smaller than the stored one —
  client behavior.
- (§3) Defining how an application protocol reacts to the absence of the
  max_datagram_frame_size transport parameter — application-protocol
  concern.
- (§5.1) Defining identifiers that multiplex flows of datagrams inside
  Datagram Data — application-protocol responsibility.
- (§5.2) Notifying the sender application that a datagram was acknowledged
  — optional (MAY) feature not shipped.
- (§5.4) A sending expiration time for congestion-delayed datagrams —
  optional feature not shipped.
- (§6) Defining a profile for acceptable use of DATAGRAM frames in 0-RTT
  (RFC 9001 §5.6) — application-protocol concern.
