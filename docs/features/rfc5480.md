# RFC 5480 — Elliptic Curve Cryptography Subject Public Key Information

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc5480.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Scope: what this TLS server SDK's certificate stack consumes/produces — EC
SubjectPublicKeyInfo parsing and encoding for P-256 and P-384. Status as of
2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 13/20 tested, 1 indirect, 6 untested.**

## §2.2 Subject Public Key

- [x] 5480-013 (§2.2) The implementation shall decode the subjectPublicKey
  BIT STRING value as an ECPoint OCTET STRING, where the BIT STRING's
  leading unused-bits octet is 0x00 and the remaining octets are the point.
  - test: `tests/crypto/ec_pubkey_test.c` — `test_ec_pubkey_extract`
  - test: `tests/crypto/spki_test.c` — `test_spki_golden`
- [x] 5480-014 (§2.2) The implementation shall support the uncompressed
  form of the ECC public key, indicated by a leading octet of 0x04.
  - test: `tests/crypto/ec_pubkey_test.c` — `test_ec_pubkey_extract`
  - test: `tests/crypto/p384chain_test.c` — `test_p384_ec_pubkey384`
- [x] 5480-015 (§2.2) If the first octet of the ECPoint OCTET STRING is
  neither 0x04 (uncompressed) nor 0x02/0x03 (compressed), then the
  implementation shall reject the public key.
  - test: `tests/crypto/ec_pubkey_test.c` — `test_ec_pubkey_bad`
- [ ] 5480-016 (§2.2) Where the compressed form of the ECC public key
  (leading octet 0x02 or 0x03) is used, the implementation shall support
  decoding it.
  - gap: `quic_x509_ec_pubkey`/`quic_x509_ec_pubkey384` accept only the
    66-byte and 98-byte uncompressed forms; a compressed-point input is
    rejected by `test_ec_pubkey_bad`, not decoded.
- [x] 5480-017 (§2.2) The implementation shall reject an ECPoint encoding
  whose length does not match the expected uncompressed-point size for the
  named curve.
  - test: `tests/crypto/ec_pubkey_test.c` — `test_ec_pubkey_bad`
  - test: `tests/crypto/p384chain_test.c` — `test_p384_ec_pubkey384`

## §3 Key Usage Bits

- [ ] 5480-020 (§3) Where the keyUsage extension is present on a
  certificate whose SubjectPublicKeyInfo indicates id-ecDH or id-ecMQV, the
  implementation shall require keyAgreement and shall reject
  digitalSignature, nonRepudiation, keyTransport, keyCertSign, or cRLSign.
  - gap: the SDK never emits or validates keyUsage against the
    id-ecDH/id-ecMQV restriction; those algorithm identifiers are not
    implemented at all (see Out of scope).

## Out of scope

Curves, algorithms, and CA-policy semantics this server SDK does not
implement, excluded from the coverage denominator:

- (§2.1.1) id-ecDH and id-ecMQV restricted key-agreement algorithm
  identifiers — the SDK has no ECDH/ECMQV key-agreement code path in the
  certificate stack; only id-ecPublicKey (unrestricted, used with ECDSA) is
  produced/consumed.
- (§2.1.1.1) The thirteen named curves other than secp256r1/secp384r1
  (secp192r1, sect163k1, sect163r2, secp224r1, sect233k1, sect233r1,
  sect283k1, sect283r1, sect409k1, sect409r1, secp521r1, sect571k1,
  sect571r1) — the SDK implements only P-256 and P-384 field/point
  arithmetic.
- (§3) Key usage bit semantics (digitalSignature, nonRepudiation,
  keyAgreement, keyCertSign, cRLSign, encipherOnly, decipherOnly) — CA
  policy guidance for certificate issuance, not a wire-format or transport
  requirement this server SDK enforces.
- (§4) Security-strength recommendations correlating key size, hash
  algorithm, and curve choice — operator/CA policy guidance, not
  implementable code behavior.
- (§5) PER (Packed Encoding Rules) encoding considerations — the SDK uses
  DER exclusively; PER is out of scope for X.509.
- Appendix A ASN.1 module entries for MD2, MD5, SHA-1, DSA, and
  Diffie-Hellman (non-EC) algorithms carried in this module for
  completeness — not ECC and not implemented by this SDK's certificate
  stack.
