# RFC 8999 — Version-Independent Properties of QUIC

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8999.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 12/15 tested, 3 indirect, 0 untested.**

## §5 QUIC Packets

- [x] 8999-001 (§5) The implementation shall classify a packet whose first
  byte has the most significant bit set as a long-header packet, and one
  with that bit cleared as a short-header packet.
  - test: `tests/transport/header_test.c` — `test_header_parse_long`
  - test: `tests/transport/header_test.c` — `test_header_parse_short`

## §5.1 Long Header

- [x] 8999-002 (§5.1) The implementation shall parse a long header as the
  first byte, a 32-bit Version, an 8-bit Destination Connection ID Length
  followed by that many DCID bytes, an 8-bit Source Connection ID Length
  followed by that many SCID bytes, and version-specific data.
  - test: `tests/transport/header_test.c` — `test_header_parse_long`
  - test: `fuzz/fuzz_header.c` — `LLVMFuzzerTestOneInput`
- [x] 8999-003 (§5.1) When building a long header, the implementation shall
  emit the same invariant layout, such that parsing reproduces the input.
  - test: `tests/transport/header_test.c` — `test_header_build_roundtrip`
  - test: `tests/transport/lhdr_build_test.c` — `test_lhdr_initial_layout`
- [x] 8999-004 (§5.1) If a long header is truncated (the Version field or a
  declared connection ID extends past the datagram), then the implementation
  shall reject the packet without reading past the buffer.
  - test: `tests/transport/header_test.c` — `test_header_truncated`
  - test: `tests/transport/header_test.c` —
    `test_header_long_too_short_for_version`

## §5.2 Short Header

- [x] 8999-005 (§5.2) The implementation shall place the Destination
  Connection ID immediately after the first byte of a short header, with no
  Version, length, or Source Connection ID fields.
  - test: `tests/transport/header_test.c` — `test_header_parse_short`
  - test: `tests/transport/packet2_test.c` — `test_short_build`
- [x] 8999-006 (§5.2) The implementation shall not read the short-header
  DCID length from the wire; the length is supplied by the receiving
  endpoint from its own connection state.
  - test: `tests/transport/header_test.c` — `test_header_parse_short`
  - test: `fuzz/fuzz_header.c` — `LLVMFuzzerTestOneInput`

## §5.3 Connection ID

- [x] 8999-007 (§5.3) The server shall use the Destination Connection ID,
  compared as an opaque length-sensitive byte string, to deliver each packet
  to the connection for which it is intended.
  - test: `tests/transport/demux_test.c` — `test_demux`
  - test: `tests/transport/conntable_test.c` — `test_conntable`

## §5.4 Version

- [x] 8999-008 (§5.4) The implementation shall treat the Version field value
  0x00000000 as reserved for version negotiation and never as a usable
  version.
  - test: `tests/transport/packet2_test.c` — `test_vneg_roundtrip`
  - test: `tests/transport/packet2_test.c` — `test_vneg_bad`

## §6 Version Negotiation

- [x] 8999-009 (§6) When a long-header datagram arrives with a version the
  server does not support, the server shall send a Version Negotiation
  packet in response.
  - test: `tests/app/srvboot_version_test.c` —
    `test_vneg_owed_for_alien_version`
- [x] 8999-010 (§6) The server shall not send a Version Negotiation packet
  in response to a packet whose version it supports.
  - test: `tests/app/srvboot_version_test.c` — `test_vneg_not_owed_for_v2`
- [~] 8999-011 (§6) The server shall not send a Version Negotiation packet
  in response to a short-header packet.
  - evidence: the VN gate (`srvboot_vn_sized` in
    `src/app/http3/server/srvboot/srvboot.c`) requires the long-header bit
    before answering; no dedicated short-header unit test.
- [x] 8999-012 (§6) The server shall build a Version Negotiation packet with
  the high bit of the first byte set, the Version field set to 0x00000000,
  both connection IDs, and a list of Supported Version fields and nothing
  else.
  - test: `tests/transport/packet2_test.c` — `test_vneg_roundtrip`
- [~] 8999-013 (§6) When receiving a Version Negotiation packet, the
  implementation shall ignore the seven Unused bits of the first byte.
  - evidence: `quic_vneg_parse` (`src/transport/packet/header/packet/vneg.c`)
    inspects only the header-form bit of byte0 (`vneg_head_ok`), so the
    Unused bits never affect parsing; no dedicated test.
- [~] 8999-014 (§6) The implementation shall ignore a Version Negotiation
  packet that contains no Supported Version fields or a truncated Supported
  Version value.
  - evidence: `quic_vneg_parse` requires one or more whole 4-byte versions
    (`versions_whole`) and the builder refuses an empty list
    (`tests/transport/packet2_test.c` — `test_vneg_bad`); the parse side of
    an empty or truncated list has no dedicated unit test.
- [x] 8999-015 (§6) When sending a Version Negotiation packet, the server
  shall copy the received Source Connection ID into the Destination
  Connection ID field and the received Destination Connection ID into the
  Source Connection ID field.
  - test: `tests/transport/packet2_test.c` — `test_vneg_respond_swap`

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

  documents — spec authorship, not server code.
- (§4) Notational conventions — descriptive only.
- (§6) Changing to a new version after receiving a Version Negotiation
  packet — client behavior; this server ignores received Version
  Negotiation packets (RFC 9368 §2.1).

  before attempting a different version — client behavior; the server half
  of downgrade prevention is covered by the RFC 9368 ledger.
- (§7) Middlebox observation and trait-ossification considerations —
  informational.
- (Appendix A) The list of incorrect assumptions about QUIC version 1 —
  informational.
