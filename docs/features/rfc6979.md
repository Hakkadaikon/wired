# RFC 6979 — Deterministic Usage of DSA and ECDSA

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc6979.txt`, not in git), scoped to the P-256 / SHA-256
instantiation the SDK ships (`src/crypto/asymmetric/ecc/p256sign/`, where
hlen == qlen == 256 bits). Each requirement carries the test that
demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 16/21 tested, 2 indirect, 3 untested.**

## §2.3 Integer Conversions

- [x] 6979-001 (§2.3.2) The implementation shall convert a bit string to an
  integer with bits2int: keep the qlen leftmost bits and read them as a
  big-endian integer (for hlen == qlen == 256 this is a plain big-endian
  read).
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_bytes`
- [x] 6979-002 (§2.3.3) The implementation shall convert an integer modulo q
  to an octet string with int2octets: rlen = 256 bits of big-endian
  encoding.
  - test: `tests/crypto/p256_field_test.c` — `test_p256_field_bytes`
- [x] 6979-003 (§2.3.4) The implementation shall compute bits2octets(b) as
  int2octets(bits2int(b) mod q), where the reduction is at most one
  conditional subtraction.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`

## §2.4 Signature Generation

- [x] 6979-004 (§2.4) The implementation shall compute the message
  representative as h = bits2int(H(m)) mod q.
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`
- [x] 6979-005 (§2.4) The implementation shall compute r as the X coordinate
  of the point kG, converted to an integer and reduced modulo q.
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`
- [x] 6979-006 (§2.4) The implementation shall compute
  s = (h + x*r) / k mod q, forming the signature pair (r, s).
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`

## §3.1 Building Blocks

- [x] 6979-007 (§3.1.1) The implementation shall use HMAC instantiated with

  - test: `tests/crypto/hmac_test.c` — `test_hmac_vectors`

## §3.2 Generation of k

- [x] 6979-008 (§3.2 b) The implementation shall initialize V to 32 octets
  of value 0x01.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-009 (§3.2 c) The implementation shall initialize K to 32 octets
  of value 0x00.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-010 (§3.2 d) The implementation shall set
  K = HMAC_K(V || 0x00 || int2octets(x) || bits2octets(h1)).
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-011 (§3.2 e) The implementation shall then set V = HMAC_K(V).
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-012 (§3.2 f) The implementation shall set
  K = HMAC_K(V || 0x01 || int2octets(x) || bits2octets(h1)), with the
  internal octet 0x01.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-013 (§3.2 g) The implementation shall then set V = HMAC_K(V).
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-014 (§3.2 h) The implementation shall generate the candidate by
  setting V = HMAC_K(V) and taking k = bits2int(V) (a single block covers
  qlen for SHA-256/P-256).
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [~] 6979-015 (§3.2 h) If the candidate k is outside [1, q-1], then the
  implementation shall set K = HMAC_K(V || 0x00), V = HMAC_K(V), and
  generate a new candidate.
  - test: `tests/crypto/rfc6979_test.c` —
    `test_rfc6979_k_in_range_boundaries` (exercises `ps_k_in_range`
    directly at k=0/n/n-1)
  - gap: the retry branch (`ps_gen_candidate`) itself is not forced to
    fire — an out-of-range first candidate has probability ~2^-32.
- [~] 6979-016 (§3.2) The implementation shall compare the candidate
  against q rather than reduce it modulo q, to avoid bias.
  - evidence: `ps_k_in_range` rejects and retries instead of reducing; the
    A.2.5 vector only exercises the accept path.

## §3.4 Usage Notes

- [ ] 6979-017 (§3.4) If a candidate k leads to r = 0, then the

  - gap: `quic_p256sign_sign` returns failure instead of looping; the case

## Appendix A.2.5 ECDSA, 256 Bits (Prime Field)

- [x] 6979-019 (A.2.5) For the published P-256 private key, SHA-256, and
  message "sample", the generated nonce k shall equal
  A6E3C57D...3D8AAD60.
  - test: `tests/crypto/rfc6979_test.c` — `test_rfc6979_sample_k`
- [x] 6979-020 (A.2.5) For the same key and message "sample", the signature
  shall match the published r = EFD48B2A...4EAF3716 and s (modulo the
  low-S normalization s -> n - s the SDK applies).
  - test: `tests/crypto/p256sign_test.c` — `test_p256sign_known_vector`
- [~] 6979-021 (A.2.5) For the same key and message "test" with SHA-256,
  the signature shall match the published (k, r, s).
  - evidence: "test" is signed and verified in the round-trip
    (`test_p256sign_ps_roundtrip`), but its k/r/s are not pinned to the
    published vector.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2.1–2.2, §2.4) Plain DSA key parameters, key pairs, and signatures —
  the SDK ships no DSA.
- (§2.4) DER encoding of the (r, s) pair — informative aside here; the TLS
  layer's `ecdsasig` encoder is tested separately.
- (§3.3) Alternate HMAC_DRBG description of the generation of k —
  informative restatement of §3.2.
- (§3.5) Rationale — informative.
- (§3.6) Variant constructions (extra entropy input k', separate secret,
  distinct HMAC hash) — deliberately not shipped; the SDK follows the base
  algorithm so the published vectors stay applicable.
- (§4) Private key generation, storage, and disposal guidance — operator
  concern.
- (A.1) Detailed K-163 example — binary-field curve the SDK lacks.
- (A.2.1–A.2.4, A.2.6–A.2.17) Vectors for DSA and for curves other than
  P-256 — not shipped (P-384 signing is not implemented; P-384 is
  verify-only for certificate chains).
- (A.2.5) SHA-1/SHA-224/SHA-384/SHA-512 rows — the SDK signs only
  ecdsa_secp256r1_sha256.
- (A.3) Sample code — informative.
