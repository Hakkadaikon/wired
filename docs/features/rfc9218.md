# RFC 9218 — Extensible Prioritization Scheme for HTTP

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9218.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger covers the server side over HTTP/3: the priority parameters
(urgency / incremental), the Priority request header field, and the HTTP/3
PRIORITY_UPDATE frame — HTTP/2-only, client, and intermediary behavior is
grouped under Out of scope.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 11/21 tested, 1 indirect, 9 untested.**

## §4 Priority Parameters

- [x] 9218-001 (§4) When receiving an HTTP request that does not carry the
  urgency or incremental priority parameters, the server shall act as if
  their default values were specified.
  - test: `tests/app/priority_test.c` — `test_priority_init_defaults`
  - test: `tests/app/h3reqdrive_test.c` — `test_reqdrive_priority_header`
- [x] 9218-002 (§4) The server shall parse the set of priority parameters
  as a Dictionary of key-value pairs.
  - test: `tests/app/priupdate_test.c` — `test_priupdate_sfv_parse`
- [x] 9218-003 (§4) If a priority parameter is unknown, carries an
  out-of-range value, or a value of an unexpected type, then the server
  shall ignore it.
  - test: `tests/app/priupdate_test.c` — `test_priupdate_sfv_parse`

## §4.1 Urgency

- [x] 9218-004 (§4.1) The server shall accept the urgency (u) parameter as

- [x] 9218-005 (§4.1) The server shall treat a smaller urgency value as a

## §7 The PRIORITY_UPDATE Frame

- [x] 9218-008 (§7) When a priority parameter is omitted from a
  PRIORITY_UPDATE frame's Priority Field Value, the server shall use that
  parameter's default value (the frame carries a complete set).
  - test: `tests/app/priupdate_test.c` — `test_priupdate_sfv_parse`
- [x] 9218-009 (§7) If the Priority Field Value fails to parse, then the
  server shall fall back to the default parameters (the optional
  H3_GENERAL_PROTOCOL_ERROR connection error is not taken).
  - test: `tests/app/priupdate_test.c` — `test_priupdate_sfv_parse`
- [x] 9218-010 (§7) When a PRIORITY_UPDATE frame references a request
  stream that is not yet open, the server should buffer the most recently
  received frame and apply it once the referenced stream is opened.
  - test: `tests/app/srvloop_priupdate_test.c` —
    `test_srvloop_priupdate_buffers_for_unopened_stream`
- [x] 9218-011 (§7.2) The server shall parse the HTTP/3 PRIORITY_UPDATE
  frame — type 0xF0700 (request) or 0xF0701 (push), the Prioritized
  Element ID, and the Priority Field Value.
  - test: `tests/app/priupdate_test.c` — `test_priupdate_roundtrip`
- [x] 9218-012 (§7.2) If a PRIORITY_UPDATE frame is truncated or carries a
  different frame type, then the server shall not decode it.
  - test: `tests/app/priupdate_test.c` — `test_priupdate_rejects_malformed`
- [x] 9218-013 (§7.2) If a PRIORITY_UPDATE frame is received on a stream
  other than the client control stream, then the server shall treat it as
  a connection error of type H3_FRAME_UNEXPECTED.
  - test: `tests/app/h3srv_priupdate_test.c` —
    `test_h3srv_priupdate_wrong_stream_unexpected`
  - test: `tests/app/srvloop_priupdate_test.c` —
    `test_srvloop_priupdate_on_request_stream_unexpected`
- [~] 9218-014 (§7.2) If a request-stream PRIORITY_UPDATE (0xF0700)
  references a stream ID that is not a request stream or is beyond the
  client-initiated bidirectional stream limit, then the server shall treat
  it as a connection error of type H3_ID_ERROR.
  - test: `tests/app/h3srv_priupdate_test.c` —
    `test_h3srv_priupdate_bad_id_error`
  - test: `tests/app/srvloop_priupdate_test.c` —
    `test_srvloop_priupdate_bad_id_error`
  - gap: the "not a request stream" (low two bits) case is checked; the
    "beyond the client-initiated bidirectional stream limit" case is not.
- [ ] 9218-015 (§7.2) If a push-stream PRIORITY_UPDATE (0xF0701) carries a
  push ID greater than the maximum push ID or one not yet promised, then
  the server shall treat it as a connection error of type H3_ID_ERROR.
  - gap: server push is not shipped, so any received push ID is
    unpromised; unenforceable until a frame consumer exists (9218-010).
- [~] 9218-016 (§7.2) The server shall not send PRIORITY_UPDATE frames of
  either type.
  - evidence: no production server code calls `quic_h3_priupdate_put`
    (encode exists for tests/clients only); no test asserts the absence.

## §10 Server Scheduling

- [ ] 9218-017 (§10) The server should send higher-urgency responses
  before lower-urgency responses when possible.
  - gap: the urgency comparator (`quic_h3_priority_higher`) is implemented
    and tested, but no response scheduler consumes it yet.
- [ ] 9218-018 (§10) While incremental responses of the same urgency are
  in flight, the server should share bandwidth among them.
- [ ] 9218-019 (§10) The server should serve non-incremental responses of
  the same urgency in ascending order of stream ID.
- [ ] 9218-020 (§10) The server should avoid starvation when scheduling a
  mix of incremental and non-incremental responses at one urgency level.
- [ ] 9218-021 (§10.1) The server should allocate some amount of bandwidth
  to streams acting as tunnels.

## Out of scope

Client-only, HTTP/2-only, intermediary, and operator-level requirements
this HTTP/3 server SDK does not implement, excluded from the coverage
denominator:

- (§2.1) SETTINGS_NO_RFC7540_PRIORITIES and HTTP/2 priority-signal
  handling — HTTP/2 only; wired serves HTTP/3.
- (§2.1.1) Sending both signal types until the server's SETTINGS arrive —
  client behavior.
- (§5) Generating a Priority response header field and controlling its
  cacheability — an origin-application / operator choice, not SDK code.
- (§7.1) The HTTP/2 PRIORITY_UPDATE frame and its stream-state rules —
  HTTP/2 only.
- (§8) Merging client- and server-driven priority parameters — behavior of
  a forwarding intermediary.
- (§9) Local scheduling of the requests a client initiates — client
  behavior.
- (§10) Server-push scheduling and Priority in PUSH_PROMISE — server push
  is not shipped.
- (§10.1, §13.1–§13.3) Backend-connection blocking, coalescing fairness,
  HTTP/1.x back ends, and intentional unfairness — intermediary / CDN
  deployment concerns.
- (§12) Considering stream priority in transport retransmission and probe
  scheduling — non-mandatory transport-layer consideration.
