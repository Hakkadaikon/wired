# RFC 9312 — Manageability of the QUIC Transport Protocol

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9312.txt`, not in git). RFC 9312 is Informational: it is
guidance for network operators and middlebox/NAT/firewall implementors on
how to observe, manage, and reason about QUIC traffic, not a set of
normative requirements on a QUIC endpoint. The document explicitly frames
itself as "primarily considering the unencrypted part of QUIC's wire image"
from the point of view of an in-network observer, and most of it (traffic
identification heuristics, NAT/firewall state timeouts, DDoS mitigation,
ECMP/QoS routing, ICMP handling, path-MTU guidance for network operators)
describes network-side behavior this server SDK does not implement. The
one place this document describes an endpoint-side behavioral choice —
the server's participation in latency spin bit signaling — has dedicated
code and tests. Everything else is grouped under Out of scope by section
family. Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 2/4 tested, 1 indirect, 1 untested.**

## §3.8.2 Using the Spin Bit for Passive RTT Measurement

- [x] 9312-001 (§3.8.2) Where the server participates in spin bit
  signaling, the server shall set the outgoing spin bit to the value of
  the spin bit observed on the peer's highest-numbered received packet
  (reflecting, rather than inverting as a client does).
  - test: `tests/transport/spin_test.c` — `test_spin_roles`
  - test: `tests/transport/spin_test.c` — `test_spin_cycle`
- [x] 9312-002 (§3.8.2) The server shall expose the spin bit only in the
  short header, keeping it absent from long-header packets.
  - test: `tests/transport/observable_test.c` — `test_observable_short`
  - test: `tests/transport/observable_test.c` — `test_observable_long`
- [~] 9312-003 (§3.8.2) The server shall base each passive RTT sample only
  on an observed edge (a change in spin value) while spin participation is
  enabled, so that a non-participating or constant-spin connection yields
  no sample.
  - test: `tests/transport/rttobs_test.c` — `test_rttobs_edge`
  - test: `tests/transport/rttobs_test.c` — `test_rttobs_sample`
  - evidence: these tests cover the sample-admission predicate
    (`quic_rttobs_sample_ok`) used to reason about what an on-path
    observer could infer; they do not exercise a live spin-bit-driven RTT
    estimator inside this SDK, since the SDK is the endpoint generating
    the signal, not the observer consuming it.
- [ ] 9312-004 (§3.8.2) The server shall randomly disable spin bit
  participation (using a fixed or randomly-set bit instead of the
  reflect/invert rule) for at least one eighth of connections, even when
  spin participation is otherwise enabled by default.
  - gap: `quic_spin_outgoing` (`src/transport/conn/cid/spin/spin.c`)
    unconditionally reflects the peer's bit for a server on every
    connection; no per-connection random opt-out of spinning is
    implemented.

## Out of scope

Network-operator- and on-path-observer-facing guidance this server SDK
does not implement or test, excluded from the coverage denominator:

- (§1 Introduction) Framing of the document's scope as network management
  guidance and a pointer to RFC 9065 on manageability of encrypted
  transports generally — not an implementable requirement.
- (§2, §2.1–§2.8 Features of the QUIC Wire Image) Description of what
  header fields (version, connection IDs, fixed bit, header type, length,
  token, packet number, key phase), coalesced packets, and port usage are
  observable in the wire image — descriptive of the wire format already
  covered by the RFC 9000/RFC 9001 ledgers; this document adds no new
  endpoint requirement.
- (§3.1, §3.1.1, §3.1.2 Identifying QUIC Traffic) Heuristics an on-path
  observer might use to recognize QUIC traffic, negotiated version, or
  garbage first packets — observer-side behavior, not server behavior.
- (§3.2 Connection Confirmation) Description of how an on-path observer
  detects handshake completion — observer-side inference, not server
  behavior.
- (§3.3 Distinguishing Acknowledgment Traffic) Discussion of why
  ACK-only-packet heuristics fail against QUIC's encrypted payload —
  observer-side limitation, not server behavior.
- (§3.4, §3.4.1 Server Name Indication) Guidance for extracting SNI from
  an unencrypted ClientHello by an on-path observer, including manual
  Initial-secret derivation and CRYPTO frame reassembly — this is
  observer/middlebox code, not server-endpoint behavior; the server's own
  TLS/SNI handling is covered by the RFC 8446/RFC 9001 ledgers.
- (§3.5 Flow Association, §3.6 Flow Teardown, §3.7 Flow Symmetry
  Measurement) Discussion of how an on-path device can (or cannot) use
  the connection ID or traffic symmetry to associate or tear down flow
  state — observer/middlebox behavior.
- (§3.8, §3.8.1 Round-Trip Time Measurement, Measuring Initial RTT)
  Discussion of how a passive network observer infers handshake RTT from
  flight timing — observer-side measurement technique, not server
  behavior.
- (§4.1 Passive Network Performance Measurement and Troubleshooting)
  Guidance for network operators on what performance metrics are passively
  observable — operator-side.
- (§4.2 Stateful Treatment of QUIC Traffic) Recommended NAT/firewall state
  timeout values and warnings against using the connection ID as a
  flow key for stateful middleboxes — middlebox/NAT configuration
  guidance, not server behavior.
- (§4.3 Address Rewriting to Ensure Routing Stability) Discussion of why
  middlebox address rewriting based on connection ID is harmful —
  middlebox guidance.
- (§4.4 Server Cooperation with Load Balancers) Guidance on coordinating
  connection ID assignment with load balancers in a server pool —
  deployment-topology guidance, not single-endpoint behavior.
- (§4.5 Filtering Behavior) Guidance against admit-then-filter firewall
  patterns — middlebox/firewall configuration guidance.
- (§4.6 UDP Blocking, Throttling, and NAT Binding) Guidance on how
  networks should throttle or block UDP/QUIC traffic and on reflection-
  prone NAT source-port bindings — network-operator guidance.
- (§4.7 DDoS Detection and Mitigation) Guidance for in-network DDoS
  classification and mitigation systems, including use of Retry packets
  as a SYN-cookie analogue from the network's perspective — the server's
  own Retry generation is covered by the RFC 9000 ledger; this section's
  content is about network-side mitigation infrastructure.
- (§4.8 Quality of Service Handling and ECMP Routing) Guidance that
  network QoS/ECMP treatment should be applied per-flow rather than
  per-packet — network-operator/router guidance.
- (§4.9 Handling ICMP Messages, §4.10 Guiding Path MTU) Guidance for
  routers on forwarding ICMP PTB messages and for network operators on
  enforcing a path MTU ceiling — network-infrastructure guidance; the
  server's own DPLPMTUD behavior is covered by the RFC 8899 ledger.
- (§6 Security Considerations) General discussion of security trade-offs
  of in-network QUIC observation and mitigation cooperation — not an
  implementable endpoint requirement.
