# RFC 9308 — Applicability of the QUIC Transport Protocol

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9308.txt`, not in git). RFC 9308 is Informational: it is
guidance for application developers deciding how to use QUIC, not a set of
normative requirements on a QUIC transport implementation. Most of its
content — fallback strategy, stream-mapping advice, packetization/latency
trade-offs, port selection, DSCP/QoS guidance, versioning rollout strategy —
is advice for an application built on top of a QUIC SDK, not something the
SDK itself implements or tests. This ledger keeps only the slices where the
document describes concrete endpoint behavior this server SDK has code for:
0-RTT replay-safety policy and the keep-alive/idle-timeout trade-off of
Section 3. Everything else is grouped under Out of scope by section family.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 2/4 tested, 1 indirect, 1 untested.**

## §3 0-RTT

- [x] 9308-001 (§3.1) The server shall treat 0-RTT application data as
  possibly replayed and shall only accept it as safe when the data is
  either idempotent or covered by additional replay protection.
  - test: `tests/transport/zerortt_policy_test.c` — `test_zerortt_policy`
- [~] 9308-002 (§3.1) The server shall not provide a means of selectively
  discarding individual pieces of already-accepted 0-RTT data once the
  connection has accepted 0-RTT.
  - evidence: `quic_zerortt_safe` (`src/transport/conn/loop/manage/
    zerortt_policy.c`) gates acceptance before data is delivered; there is
    no post-acceptance selective-discard API, matching the RFC's
    description of accept-all-or-nothing 0-RTT handling.

## §3.2 Session Resumption versus Keep-Alive

- [x] 9308-003 (§3.2) The server shall compute the effective connection
  idle timeout as the minimum of the client's and server's advertised idle
  timeout values.
  - test: `tests/transport/idlenego_test.c` —
    `test_idlenego_both_nonzero_min`
  - test: `tests/transport/idlenego_test.c` — `test_idlenego_one_zero`
- [ ] 9308-004 (§3.2) The server should send PING frames as keep-alives no
  more often than roughly every 30 seconds during long idle periods, per
  the trade-off this section and RFC 8085 §3.5 describe.
  - gap: `quic_keepalive_interval`/`quic_keepalive_due`
    (`tests/transport/keepalive_test.c`) derive the keep-alive interval as
    half of the negotiated idle timeout, a related but distinct policy
    from the RFC's fixed ~30-second floor; no test pins the SDK to the
    RFC's specific interval recommendation.

## Out of scope

Application- and operator-facing guidance this server SDK does not
implement or test, excluded from the coverage denominator:

- (§2 The Necessity of Fallback) Guidance for applications on falling back
  from QUIC to TCP/TLS when UDP is blocked — an application-layer decision
  outside the transport SDK.
- (§4 Use of Streams, §4.1–§4.5) Advice on mapping application data to
  streams, prioritization, ordered/reliable delivery assumptions, flow
  control deadlock avoidance, and stream limit management — these are
  design guidelines for protocols built on top of QUIC streams (e.g.
  HTTP/3), not transport-level requirements on this SDK.
- (§5 Packetization and Latency) Discussion of frame bundling/coalescing
  delay trade-offs and padding for traffic-analysis resistance — advisory
  for applications tuning latency vs. bandwidth, not a testable SDK
  requirement.
- (§6 Error Handling) Description of QUIC's existing error-signaling
  surface (CONNECTION_CLOSE, STOP_SENDING, RESET_STREAM) for application
  protocol designers — already covered by the RFC 9000 ledger.
- (§7 Acknowledgment Efficiency) Discussion of alternative ACK-frequency
  strategies and a pointer to the ACK Frequency extension draft —
  descriptive, not a requirement on this SDK.
- (§8, §8.1 Port Selection and Application Endpoint Discovery) Guidance on
  UDP port registration conventions and avoiding source ports associated
  with reflection-attack-prone services — deployment/operational choice,
  not transport code.
- (§9 Connection Migration) Discussion of when and why client-initiated
  migration helps applications — the transport mechanics themselves are
  covered by the RFC 9000 ledger; this section is applicability
  discussion for the application layer.
- (§10 Connection Termination) General discussion of QUIC's termination
  modes for application designers — the mechanisms are covered by the RFC
  9000 ledger.
- (§11, §11.1–§11.3 Information Exposure and the Connection ID) Deployment
  guidance for load-balanced server pools, connection ID linkability
  mitigation, and Retry-based redirection — operator/deployment-topology
  concerns, not single-endpoint SDK behavior.
- (§12 QoS and DSCP) Guidance to keep a single DSCP per connection and to
  avoid excessive parallel connections for differentiated treatment —
  network-operator and application deployment guidance.
- (§13 Use of Versions and Cryptographic Handshake, §14 Enabling
  Deployment of New Versions) Discussion of version selection policy and
  a three-stage rollout strategy for new QUIC versions across a server
  fleet — deployment-process guidance, not implementable endpoint logic.
- (§15 Unreliable Datagram Service over QUIC) Pointer to RFC 9221 and
  discussion of when applications should prefer the datagram extension
  over streams — covered by the RFC 9221 ledger where implemented.
- (§17 Security Considerations) General pointer to RFC 9000/RFC 9001
  security considerations and fallback security expectations — already
  covered by those RFCs' own ledgers.
