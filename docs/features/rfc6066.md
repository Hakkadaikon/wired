# RFC 6066 — TLS Extensions: Extension Definitions

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc6066.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger is scoped to Server Name Indication (§3) as this server SDK
handles it; the other extensions this RFC defines (max_fragment_length,
client_certificate_url, trusted_ca_keys, truncated_hmac, status_request) are
grouped under Out of scope — `src/` ships none of them (grep-verified).
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 7/13 tested, 3 indirect, 3 untested.**

## §3 Server Name Indication

- [x] 6066-001 (§3) The server shall accept a ClientHello "server_name"
  extension whose extension_data is a ServerNameList of one host_name
  entry: name_type(1)=0, a 16-bit length, and the hostname bytes.
  - test: `tests/tls/sni_test.c` — `test_sni_roundtrip`
  - test: `tests/tls/sni_extract_test.c` —
    `test_sni_extract_from_clienthello`
- [x] 6066-002 (§3) If the server_name extension's name_type is not
  host_name(0), then the server shall reject the entry.
  - test: `tests/tls/sni_test.c` — `test_sni_wrong_type`
  - test: `tests/tls/sni_extract_test.c` —
    `test_sni_extract_wrong_name_type`
- [x] 6066-003 (§3) If a length field inside the server_name extension
  overruns the available bytes, then the server shall reject the
  extension rather than over-read.
  - test: `tests/tls/sni_test.c` — `test_sni_truncated`
  - test: `tests/tls/sni_extract_test.c` — `test_sni_extract_truncated`
- [x] 6066-004 (§3) The server shall locate the server_name extension
  (extension_type 0x0000) inside a ClientHello's extensions block by its
  type, independent of the other extensions present.
  - test: `tests/tls/ch_ext_test.c` — `test_ch_ext_finds_alpn_and_sni`
- [~] 6066-005 (§3) The ServerNameList MUST NOT contain more than one name
  of the same name_type.
  - evidence: `quic_tls_sni_decode`/`quic_salpn_extract_sni` read only the
    first (and only supported) host_name entry; the SDK never builds or
    accepts a list with a second entry of the same type, so the
    prohibition holds by construction — no dedicated multi-entry rejection
    test.
- [ ] 6066-006 (§3) If the server understood the server_name extension but
  does not recognize the server name, the server should either abort the
  handshake with a fatal unrecognized_name(112) alert or continue the
  handshake.
  - gap: the extracted SNI is not yet wired to certificate selection or
    an unrecognized_name alert path in `src/`; `quic_salpn_extract_sni` is
    exercised directly by tests but has no caller in `src/`.
- [x] 6066-007 (§3) A server that receives a client hello containing the
  server_name extension may use it to guide certificate selection, and in
  that event shall include an extension of type server_name (with empty
  extension_data) in the server hello.
  - test: `tests/tls/tlsdriver_sni_test.c` — `test_tlsdriver_sni_present`
  - gap: this is exercised on the client-emitting side
    (`quic_tlsdriver_set_sni`/`quic_tlsdriver_raw_client_hello`); the
    server-hello echo-back is not implemented in `src/`.
- [x] 6066-008 (§3) Currently, the only server name supported is the DNS
  hostname type; the associated data structure is a variable-length
  vector beginning with a 16-bit length.
  - test: `tests/tls/sni_test.c` — `test_sni_roundtrip`
- [x] 6066-009 (§3) The hostname is represented as a byte string using
  ASCII encoding without a trailing dot, and DNS hostnames are compared
  case-insensitively.
  - test: `tests/crypto/san_test.c` — `test_san_case_fold`
  - evidence: SNI itself is carried opaquely (`quic_tls_sni_encode`/
    `quic_tls_sni_decode` do not case-fold); case-insensitive comparison
    is demonstrated at the hostname-matching layer (RFC 6125 6.4.1), the
    consumer of the extracted SNI value.
- [ ] 6066-010 (§3) A server that implements resumption using the
  server_name extension shall not accept a resumption request if the
  server_name extension contains a name different from the one
  associated with the session being resumed, and shall not include a
  server_name extension when resuming a session.
  - gap: session resumption (`tests/tls/resume_test.c`) does not key on or
    compare the server_name extension.

## §9 Error Alerts / §11.1 Security Considerations for server_name

- [~] 6066-011 (§9, §11.1) Implementations shall ensure that a buffer
  overflow does not occur, whatever the values of the length fields in
  server_name.
  - evidence: `quic_tls_sni_decode`/`quic_salpn_extract_sni` bound every
    length field against the input span before indexing
    (`3 + len <= buf.n`, `2 + list_len <= sni_ext.n`), verified by
    `tests/tls/sni_test.c test_sni_truncated` and
    `tests/tls/sni_extract_test.c test_sni_extract_truncated` (out-of-range
    lengths return 0 rather than reading past the buffer); no dedicated
    fuzz/ASAN-style overflow probe.
- [~] 6066-012 (§11.1) Since a client can present a different server_name
  at the application protocol layer, an application relying on the two
  being the same must check that the client did not present a different
  name at the application layer.
  - evidence: this is an application-layer (caller) obligation outside
    the SDK's TLS extension code; the SDK exposes the decoded SNI view for
    the caller to compare, per `docs/security.md`'s "Caller
    responsibility" note on hostname checks — no SDK-internal test
    applies.
- [ ] 6066-013 (§9) The new "unrecognized_name" alert (112), sent only
  after the sending party has received an extended hello from its peer,
  is defined for use when the server does not recognize the presented
  server name.
  - gap: no unrecognized_name(112) alert code exists in `src/`.

## Out of scope

Extensions this RFC defines that this server SDK does not implement
(grep-verified absent from `src/`), excluded from the coverage denominator:

- (§4) Maximum Fragment Length Negotiation — no `max_fragment_length`
  extension handling exists; QUIC's own datagram/packet sizing (RFC 9000
  §14, RFC 8899) governs record sizing instead.
- (§5) Client Certificate URLs (the `client_certificate_url` extension and
  the `CertificateURL`/`CertificateStatus` handshake messages) — not
  shipped; this server SDK does not perform client-certificate
  authentication via URL dereference.
- (§6) Trusted CA Indication (`trusted_ca_keys`) — not shipped.
- (§7) Truncated HMAC (`truncated_hmac`) — not shipped; QUIC 1.3's AEAD
  suites do not use HMAC-truncation-style MACs on records.
- (§8) Certificate Status Request (OCSP stapling, `status_request`) — not
  shipped.
- (§11.2–§11.6) Security considerations for the five out-of-scope
  extensions above — inapplicable once the extensions themselves are not
  implemented.
