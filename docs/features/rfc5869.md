# RFC 5869 — HMAC-based Extract-and-Expand Key Derivation Function (HKDF)

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc5869.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 4/8 tested, 1 indirect, 3 untested.**

## §2.2 Step 1: Extract

- [x] 5869-001 (§2.2) The implementation shall compute
  PRK = HMAC-Hash(salt, IKM), using the salt as the HMAC key and the input
  keying material as the HMAC message.
  - test: `tests/crypto/hkdf_test.c` — `test_hkdf_rfc5869`
- [~] 5869-002 (§2.2) When no salt is provided, the implementation shall use
  a string of HashLen zero octets as the salt.
  - evidence: the API takes an explicit salt span; HMAC's key zero-padding
    makes a zero-length salt equivalent to HashLen zeros, and the TLS 1.3
    key schedule supplies the zero salt itself. No dedicated no-salt test.

## §2.3 Step 2: Expand

- [x] 5869-003 (§2.3) The implementation shall compute OKM as the first L
  octets of T(1) | T(2) | ... | T(N), where
  T(i) = HMAC-Hash(PRK, T(i-1) | info | i) and T(0) is the empty string.
  - test: `tests/crypto/hkdf_test.c` — `test_hkdf_rfc5869`
- [x] 5869-004 (§2.3) The implementation shall append the block counter to
  each T(i) input as a single octet starting at 0x01.
  - test: `tests/crypto/hkdf_test.c` — `test_hkdf_rfc5869`
  - gap: the A.1 vector's 42-octet output spans two blocks, pinning the
    0x01 and 0x02 counters; counters beyond 0x02 are not exercised.
- [ ] 5869-005 (§2.3) If L exceeds 255*HashLen octets, then the
  implementation shall reject the request.
  - gap: the bound is enforced in `quic_hkdf_expand` (`expand_ok`), but no
    test drives L past 255*32.

## Appendix A Test Vectors

- [ ] 5869-007 (A.2) The implementation shall reproduce Test Case 2

- [ ] 5869-008 (A.3) The implementation shall reproduce Test Case 3

  - gap: also the only vector exercising a zero-length info string.

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2) HKDF instantiated with hash functions other than SHA-256 — the
  SDK's TLS 1.3/QUIC profile uses HKDF-SHA-256 only.
- (§3.1) Guidance on choosing or providing salt values — application
  design guidance; the TLS 1.3 key schedule fixes the salts.
- (§3.2) Guidance on the content of the 'info' input — application design
  guidance; RFC 8446 fixes the HkdfLabel structure.
- (§3.3) Skipping the extract step, and the SHOULD NOT for Diffie-Hellman
  inputs — the caller's key-schedule concern; RFC 8446 always extracts.
- (§3.3) NOT RECOMMENDED use of PRK directly as OKM — caller guidance.
- (§3.4) Independence of IKM and salt sources — operator/application
  concern.
- (A.4–A.7) SHA-1 test vectors — the SDK ships no SHA-1.
