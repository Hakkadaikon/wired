# RFC 9297 — HTTP Datagrams and the Capsule Protocol

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9297.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 8/23 tested, 2 indirect, 13 untested.**

## §2 HTTP Datagrams

- [ ] 9297-001 (§2) The server shall only send an HTTP Datagram in
  association with a request that explicitly supports HTTP Datagrams.
  - gap: no request-type gate exists; the SDK's only HTTP Datagram user is
    WebTransport, whose CONNECT association is implicit rather than checked
    against a per-request "supports HTTP Datagrams" flag.
- [ ] 9297-002 (§2) If an HTTP Datagram is received and it is associated
  with a request that has no known semantics for HTTP Datagrams, then the
  server shall terminate the request with H3_DATAGRAM_ERROR (0x33).
  - gap: no H3_DATAGRAM_ERROR constant or send-site exists in `src/`.

## §2.1 HTTP/3 Datagrams

- [x] 9297-003 (§2.1) The implementation shall format the QUIC DATAGRAM
  frame payload as a Quarter Stream ID variable-length integer followed by
  the HTTP Datagram Payload.
  - test: `tests/app/wtwire_test.c` — `test_wtwire_qsid_put`
  - test: `tests/app/srvrun_test.c` —
    `test_srvrun_wt_send_datagram_to_prefixes_qsid`
- [x] 9297-004 (§2.1) The implementation shall compute the Quarter Stream
  ID as the client-initiated bidirectional stream ID divided by four.
  - test: `tests/app/h3dgram_test.c` — `test_quarter_from_stream`
  - test: `tests/app/wtwire_test.c` — `test_wtwire_qsid_roundtrip`
- [ ] 9297-005 (§2.1) If a received HTTP/3 Datagram's Quarter Stream ID
  exceeds 2^60-1, then the server shall treat it as an HTTP/3 connection
  error of type H3_DATAGRAM_ERROR (0x33).
  - gap: no receive-side Quarter Stream ID parsing exists at all (see
    9297-010); this bound is unchecked.
- [ ] 9297-006 (§2.1) If a received QUIC DATAGRAM frame payload is too
  short to parse the Quarter Stream ID field, then the server shall treat
  it as an HTTP/3 connection error of type H3_DATAGRAM_ERROR (0x33).
  - gap: same root cause as 9297-005; `quic_wtwire_qsid_take` exists and is
    tested standalone (rejects a truncated varint) but has no call site in
    `src/`, so the connection-error consequence is never wired.
- [ ] 9297-007 (§2.1) The implementation shall not send an HTTP/3 Datagram
  unless the corresponding stream's send side is open.
  - gap: no send-side-open check gates `srvrun_queue_datagram` /
    `srvrun_send_datagram_now` beyond the SETTINGS_H3_DATAGRAM gate
    (9297-013).
- [ ] 9297-008 (§2.1) If an HTTP/3 Datagram is received after the
  corresponding stream's receive side is closed, then the server shall
  silently drop it.
  - gap: incoming datagrams are delivered to whichever WT slot
    `srvrun_wt_slot_for_new_stream` returns, with no per-session
    receive-side-closed check.
- [ ] 9297-009 (§2.1) When an HTTP/3 Datagram is received whose Quarter
  Stream ID maps to a stream that has not yet been created, the server
  shall either drop it silently or buffer it temporarily awaiting the
  stream's creation.
  - gap: `wired_wt_session_offer_datagram` (session.h) buffers datagrams
    pre-establishment, but that path is keyed to session state, not to a
    Quarter-Stream-ID-derived stream lookup (9297-010 is the missing link
    between the two).
- [ ] 9297-010 (§2.1) When an HTTP/3 Datagram is received whose Quarter
  Stream ID maps to a stream that cannot be created due to client-initiated
  bidirectional stream limits, the server should treat it as an HTTP/3
  connection error of type H3_ID_ERROR.
  - gap: no receive-side Quarter Stream ID demultiplexing exists;
    `srvrun_deliver_rx_datagram` (srvrun.c) delivers the raw QUIC DATAGRAM
    payload straight to the most-recent WT slot without ever calling
    `quic_wtwire_qsid_take` to recover or validate the stream ID it names.

## §2.1.1 The SETTINGS_H3_DATAGRAM HTTP/3 Setting

- [x] 9297-011 (§2.1.1) Where an endpoint is willing to receive HTTP/3
  Datagrams, the server shall send the SETTINGS_H3_DATAGRAM (0x33) setting
  with a value of 1.
  - test: `tests/app/h3settings_build_test.c` —
    `test_h3settings_build_h3_datagram_and_wt_enabled`
  - test: `tests/app/h3settings_control_settings_test.c` —
    `test_h3settings_control_settings_advertises_connect_protocol`
- [ ] 9297-012 (§2.1.1) If the SETTINGS_H3_DATAGRAM setting is received
  with a value that is neither 0 nor 1, then the server shall terminate
  the connection with error H3_SETTINGS_ERROR.
  - gap: `settings_check.c` validates reserved/allowed/unknown identifiers
    but does not range-check the SETTINGS_H3_DATAGRAM value.
- [x] 9297-013 (§2.1.1) The server shall not send a QUIC DATAGRAM frame
  until the SETTINGS_H3_DATAGRAM setting has been sent.
  - test: `tests/app/srvrun_test.c` —
    `test_srvrun_datagram_dropped_before_settings_sent`
  - test: `tests/app/srvrun_test.c` —
    `test_srvrun_wt_send_datagram_to_requires_settings_sent`
- [ ] 9297-014 (§2.1.1) When the server decides to accept 0-RTT data, the
  server shall send a SETTINGS_H3_DATAGRAM setting greater than or equal
  to the value it sent in the connection where it sent the
  NewSessionTicket message.
  - gap: no 0-RTT-resumption-specific SETTINGS_H3_DATAGRAM comparison
    exists; `control_settings.c` always advertises a fixed value
    independent of any remembered prior-session value.
- [~] 9297-015 (§2.1.1) The maximum permitted value of the
  SETTINGS_H3_DATAGRAM setting parameter is 1.
  - evidence: `settings_build.c`'s `append_h3_datagram` only ever appends
    the literal value 1 (`control_settings.c`'s `in.enable_h3_datagram = 1`
    on the one call site); no code path can emit a value other than 0
    (absent) or 1, so the bound holds by construction, but no test asserts
    it directly.
- [x] 9297-016 (§2.1.1) It is recommended that the server always send the
  SETTINGS_H3_DATAGRAM setting with a value of 1 when it supports
  receiving HTTP/3 Datagrams, to avoid sticking out.
  - test: `tests/app/h3settings_build_test.c` —
    `test_h3settings_build_h3_datagram_and_wt_enabled`

## §3 Capsules

- [~] 9297-017 (§3.2) The Capsule Protocol shall encode each Capsule as a
  Capsule Type variable-length integer, a Capsule Length variable-length
  integer, and a Capsule Value of that length.
  - test: `tests/app/capsule_test.c` — `test_capsule_roundtrip_small_type`
  - test: `tests/app/capsule_test.c` — `test_capsule_roundtrip_large_types`
  - evidence: the generic codec is fully round-trip tested; this item is
    `[~]` rather than `[x]` because the wire byte layout (type-then-length-
    then-value ordering, not just round-trip equality) is confirmed by
    inspection of `capsule.c`, not asserted byte-for-byte in the cited
    tests.
- [x] 9297-018 (§3.2) If a receiver encounters a Capsule with an unknown
  Capsule Type, then the implementation shall silently drop that Capsule
  and skip over it to parse the next Capsule.
  - test: `tests/app/wtcapsule_test.c` —
    `test_wtcapsule_no_per_stream_flow_control_capsule`
  - test: `tests/app/wtcapsule_test.c` —
    `test_wtcapsule_wrong_type_does_not_advance`
- [ ] 9297-019 (§3.2) The Capsule Protocol shall not be used with messages
  that contain Content-Length, Content-Type, or Transfer-Encoding header
  fields, and status codes 204, 205, and 206 shall not be sent on
  responses that use the Capsule Protocol.
  - gap: no header-field/status-code cross-check against Capsule Protocol
    usage exists; the SDK's only Capsule Protocol user (WebTransport)
    always responds with a bare 2xx/403/429 status built by
    `srvrun_start_wt_status`, which structurally cannot carry those headers,
    but nothing asserts the exclusion as a rule.
- [x] 9297-020 (§3.3) The implementation shall treat a Capsule payload
  that contains additional bytes after its identified fields, or one that
  terminates before the end of its identified fields, as malformed.
  - test: `tests/app/capsule_test.c` — `test_capsule_decode_truncated_value`
  - test: `tests/app/wtcapsule_test.c` —
    `test_wtcapsule_close_decode_body_too_short`
- [ ] 9297-021 (§3.3) If the receive side of a stream carrying Capsules is
  terminated cleanly and the last Capsule on the stream was truncated,
  then the server shall treat this as a malformed or incomplete message.
  - gap: no code path correlates a clean FIN on the CONNECT stream with a
    truncated trailing Capsule; WebTransport's own CONNECT-stream body is
    not fed through a Capsule stream-reassembly path in this SDK.
- [ ] 9297-022 (§3.4) The Capsule-Protocol header field shall be an Item
  Structured Field whose value is a Boolean; any other value type shall
  be handled as if the field were not present.
  - gap: no Capsule-Protocol header field is generated, parsed, or
    validated anywhere in `src/` (the draft explicitly says WebTransport
    does not require it, so this is intentionally unbuilt rather than
    missed, but it remains untested).

## Out of scope

Client-only, intermediary-only, and HTTP-version requirements this server
SDK does not implement, excluded from the coverage denominator:

- (§1) HTTP Datagram support for HTTP/1.x and HTTP/2 — this SDK is an
  HTTP/3-only server.
- (§2.2) Negotiating and conveying HTTP Datagrams via the Capsule Protocol
  as the sole transport when QUIC DATAGRAM frames are unavailable — this
  SDK always has QUIC DATAGRAM available (RFC 9221) and does not implement
  the Capsule-only fallback path for HTTP Datagrams.
- (§2.1.1) Client behavior storing the server's SETTINGS_H3_DATAGRAM value
  across 0-RTT and validating the resumed value against it — client
  behavior.
- (§3) Intermediary forwarding, re-encoding, and translation behavior for
  Capsules and HTTP Datagrams (§3.2's intermediary SHOULD-forward
  guidance, §3.5's QUIC-DATAGRAM-to-Capsule re-encoding heuristics) — this
  SDK is an endpoint, not an HTTP intermediary.
- (§3.1) HTTP/1.x-specific and HTTP/2-specific definitions of the "data
  stream" concept — this SDK is HTTP/3-only.
- (§4) Security-consideration guidance about "sticking out" and JavaScript
  Web Platform API accessibility — informational, not a testable server
  behavior.
