# RFC 8899 — Packetization Layer Path MTU Discovery for Datagram Transports

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc8899.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger covers the generic DPLPMTUD machinery a QUIC server carries —
the probe/ack/ceiling search around `quic_pmtu` — as RFC 9000 §14.3 applies
it; the TCP- and SCTP-specific material is grouped under Out of scope.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 12/31 tested, 7 indirect, 12 untested.**

## §3 Features Required to Provide Datagram PLPMTUD

- [~] 8899-001 (§3.1) The server shall not send a datagram, other than a
  probe packet, larger than the current PLPMTU.
  - evidence: the send path builds every non-probe datagram at the
    1200-byte base size (`QUIC_MAX_DATAGRAM`); a raised PLPMTU is not yet
    consulted for packet sizing, so no oversized non-probe send can occur —
    no dedicated size-cap test.
- [ ] 8899-002 (§3.2) The server shall transmit probe packets larger than
  the PLPMTU, with the IPv4 DF bit set and without network-layer endpoint
  fragmentation.
  - gap: probe transmission is not wired into the UDP send path.
- [x] 8899-003 (§3.3) When feedback indicates a probe packet was received
  by the destination, the server shall treat the probed size as delivered.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`
- [x] 8899-004 (§3.4) The server shall remain robust when probe packets
  are lost: a lost probe bounds the search without lowering the validated
  size.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_loss_caps`
- [ ] 8899-005 (§3.4) The server should use probe packets that do not
  carry user data requiring retransmission if lost.
  - gap: probe packet construction (e.g. PING plus PADDING) is not
    implemented; the state machine exists unwired.
- [ ] 8899-006 (§3.7) While probe transmission is not controlled by the
  congestion controller, the server shall keep the interval between probe
  packets at least one RTT.
- [ ] 8899-007 (§3.7) The server shall not treat loss of a probe packet as
  an indication of congestion nor trigger a congestion control reaction.
- [ ] 8899-008 (§3.7) When the PLPMTU or MPS is updated, the server shall
  not increase the congestion window measured in bytes.
- [ ] 8899-009 (§3.8) The server shall not apply flow control to probe
  packets that carry no user data.
- [~] 8899-010 (§3.5) The server should utilize information about the
  maximum packet size the sender's local link can transmit when choosing
  the search ceiling.
  - evidence: the ceiling is the fixed Ethernet-derived constant
    `QUIC_PMTU_MAX` (1452); the local interface MTU is not queried.

## §4.1 PLPMTU Probe Packets

- [x] 8899-011 (§4.1) The server shall distinguish an in-band data block
  from added padding, so padding is never passed to the application.
  - test: `tests/transport/dispatch_state_test.c` — `test_dispatch_padding`
- [~] 8899-012 (§4.1) The server shall make each probe uniquely
  identifiable within the Maximum Segment Lifetime, robust to reordering
  and replay.
  - evidence: probes would be ordinary ack-eliciting QUIC packets
    identified by their packet number (RFC 9000); no probe-specific test.

## §4.2 Confirmation of Probed Packet Size

- [x] 8899-013 (§4.2) The server shall confirm a probed packet size by the
  packetization layer's own acknowledgment of the probe, QUIC being an
  acknowledged PL (§6.3).
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_reaches_max`

## §4.3 Black Hole Detection and Reducing the PLPMTU

- [~] 8899-014 (§4.3) The server shall not rely solely on PTB messages to
  detect a black hole.
  - evidence: PTB messages are not processed at all; loss detection is
    purely acknowledgment-driven, satisfying the prohibition by
    construction.
- [ ] 8899-015 (§4.3) When the current PLPMTU is detected as no longer
  supported, the server shall set a lower PLPMTU and a lower MPS.
  - gap: the validated size only rises; black-hole-driven reduction toward
    the base is not implemented.

## §4.4 The Maximum Packet Size (MPS)

- [ ] 8899-016 (§4.4) The server should provide the application with a
  primitive that returns the MPS derived from the current PLPMTU.

## §4.5 Disabling the Effect of PMTUD

- [ ] 8899-017 (§4.5) The server shall suspend network-layer PMTU
  enforcement of outgoing packets for flows using DPLPMTUD.
  - gap: the kernel PMTU-discovery socket option is not configured.

## §5.1 DPLPMTUD Components

- [ ] 8899-018 (§5.1.1) The server shall configure the PROBE_TIMER to no
  less than 1 second (and preferably larger than 15 seconds).
  - gap: no probe timer exists; probing is not yet scheduled.
- [~] 8899-019 (§5.1.1) While an acknowledged PL is used, the server shall
  not use the CONFIRMATION_TIMER.
  - evidence: QUIC is an acknowledged PL and the module implements no
    confirmation timer; the prohibition is satisfied by absence.
- [ ] 8899-020 (§5.1.2) The server shall retry an unacknowledged probe up
  to MAX_PROBES (default 3) consecutive times before concluding the size
  is unsupported.
  - gap: a single lost probe caps the search; no PROBE_COUNT retry
    accounting exists.
- [x] 8899-021 (§5.1.2) The server shall use a BASE_PLPMTU of 1200 bytes
  as the initial size expected to work for most paths.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`
- [x] 8899-022 (§5.1.2) The server shall bound the search at a MAX_PLPMTU
  ceiling not exceeding what the outgoing interface can carry.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_reaches_max`
- [x] 8899-023 (§5.1.3) The server shall treat PROBED_SIZE as tentative,
  assigning it to the PLPMTU only after the probe is acknowledged.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`

## §5.2 State Machine

- [x] 8899-024 (§5.2) While in the Base phase, the server shall confirm
  connectivity and the BASE_PLPMTU implicitly through the connection
  handshake carried in base-sized (1200-byte-padded) datagrams.
  - test: `tests/app/srvloop_test.c` — `test_srvloop_send_initial_roundtrip`
  - test: `tests/transport/pad_test.c` — `test_pad`
- [x] 8899-025 (§5.2) While in the Searching phase, when a probe packet is
  acknowledged, the server shall set the PLPMTU to the PROBED_SIZE and
  continue the search with a larger candidate.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`
- [x] 8899-026 (§5.2) The server shall leave the Searching phase when a
  probe of the maximum size is acknowledged or when no viable larger
  candidate remains.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_reaches_max`
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_loss_caps`
- [ ] 8899-027 (§5.2) While in the Search Complete phase, the server shall
  resume the search when the PMTU_RAISE_TIMER (600 s) expires.
  - gap: no raise timer; a completed search is never reentered.

## §5.3 Search to Increase the PLPMTU

- [x] 8899-028 (§5.3.1) The server shall select probe candidates above the
  validated PLPMTU and no larger than the ceiling learned from losses.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_grow`
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_loss_caps`
- [~] 8899-029 (§5.3.2) The server should select probe sizes that maximize
  the gain in PLPMTU from each search step.
  - evidence: the search uses fixed 64-byte steps (`QUIC_PMTU_STEP`);
    `test_pmtu_grow` asserts the first step, but no test evaluates
    step-size optimality.
- [x] 8899-030 (§5.3.3) The server shall stay resilient to inconsistent
  path information by never re-probing a size the path already dropped.
  - test: `tests/transport/pmtu_test.c` — `test_pmtu_loss_caps`

## §8 Security Considerations

- [~] 8899-031 (§8) The server shall confirm probe reception by a method
  protected from off-path packet injection.
  - evidence: probe acknowledgments arrive in AEAD-protected ACK frames
    bound to the connection, which an off-path attacker cannot forge; no
    dedicated forgery test.

## Out of scope

TCP/SCTP-specific methods, PTB processing (which §4.6.1 permits omitting),
and requirements on layers this SDK does not ship, excluded from the
coverage denominator:

- (§3.6, §4.6, §6.1.6, §8) Validation and use of ICMP PTB messages — wired
  ignores PTB messages entirely, the simple-implementation option §4.6.1
  explicitly permits; nothing is derived from PTB_SIZE.
- (§3.9) Sharing the discovered PLPMTU through the IP-layer cache with
  other PL instances — cross-flow caching is not shipped.
- (§4.4) Segmenting or resegmenting data blocks larger than the MPS —
  QUIC streams already segment, and QUIC datagrams cannot be fragmented
  (RFC 9221).
- (§5.2) The ERROR phase for paths that cannot carry the BASE_PLPMTU —
  QUIC requires 1200-byte UDP payload support (RFC 9000 §14), so
  connection setup fails instead of a below-base fallback.
- (§5.2) The DISABLED state and re-entry after loss of connectivity — a
  QUIC connection that loses connectivity closes; discovery does not
  outlive it.
- (§6.1) Application-implemented DPLPMTUD over UDP or UDP-Lite — wired's
  packetization layer is QUIC, not a bare-UDP application.
- (§6.2) DPLPMTUD for SCTP, SCTP/UDP, and SCTP/DTLS — SCTP is not shipped.
- (§1.2, RFC 4821 material) PLPMTUD for TCP — TCP is not shipped.
