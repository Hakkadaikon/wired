# RFC 9114 — HTTP/3

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9114.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 56/81 tested, 9 indirect, 16 untested.**

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§3.1) HTTP/3 endpoint discovery and Alt-Svc advertisement — operator /
  deployment concern.
- (§3.1) Verifying the server certificate against the URI's origin —
  client behavior.
- (§3.1.2) Ensuring the server is willing to serve non-"https" schemes —
  client behavior.
- (§3.2) Sending SNI and falling back to TCP-based HTTP — client behavior.
- (§3.3) Revalidating certificates when reusing a connection for a new
  origin, and limiting connections per authority — client behavior.
- (§3.3) Signalling 421 (Misdirected Request) — optional origin-selection
  response left to the application handler.
- (§4.1) Closing the request stream after sending and not tying closure to
  the response — client behavior.
- (§4.1.1) Cancelling requests with H3_REQUEST_CANCELLED and the retry
  rules for cancelled requests — client behavior.
- (§4.4) CONNECT proxy tunneling to a TCP server (H3_CONNECT_ERROR,
  FIN/RST mapping, target-port restrictions) — wired is an origin server;
  CONNECT is honored only as Extended CONNECT (RFC 9220 / WebTransport).
- (§4.6) Consuming pushed responses (buffering push streams, sending
  CANCEL_PUSH, cacheability, authority validation of promises) — client
  behavior.
- (§5.1) Keeping connections open for outstanding responses and gateway
  pre-warming — client / gateway behavior.
- (§5.2) Choosing the push ID in a client-sent GOAWAY — client behavior.
- (§5.4) Assuming requests may have been processed when a connection
  terminates without GOAWAY — client behavior.
- (§6.1) Treating server-initiated bidirectional streams as
  H3_STREAM_CREATION_ERROR — client behavior.
- (§6.2.2) Push stream header checks (duplicate push ID as H3_ID_ERROR,
  not aborting before the header) — client behavior.
- (§7.2.4.2) Storing server settings and complying with them when
  attempting 0-RTT — client behavior.
- (§7.2.5) Checks on received PUSH_PROMISE frames (push ID beyond the
  advertised maximum, mismatched duplicate promises) — client behavior.
- (§7.2.6) Treating a GOAWAY with a wrong identifier type as H3_ID_ERROR —
  client behavior (the server-side ID-type rule is 9114-036).
- (§10.2, §10.4–§10.7, §10.10, §10.11) Cross-protocol attack notes,
  multi-tenant push authority, CONNECT proxy load, compression-context
  separation, padding and traffic analysis, migration-aware logging, and
  privacy considerations — informational / operator guidance.
- (§11) IANA registrations and registry policies — process requirements
  for specification authors.
- (Appendix A) HTTP/2 transition guidance — informative.
