# RFC 9000 — QUIC: A UDP-Based Multiplexed and Secure Transport

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc9000.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

## Out of scope

Client-only or operator-level requirements this server SDK does not
implement, excluded from the coverage denominator:

- (§2.3, §2.4, §5.3) Stream/connection API operation lists — informative
  API guidance, not wire behavior.
- (§5.1) Prohibition on multiplexing several zero-length-CID connections on
  one address/port — deployment configuration guidance.
- (§5.2.1) Client packet handling (zero-length-CID matching, buffering
  packets pending keys, discarding other-version packets) — client
  behavior.
- (§5.2.3) Simple load balancer continuity considerations — operator
  deployment concern.
- (§6.2) Future-version negotiation robustness — reserved for future
  Standards Track work.
- (§9.6.1, §9.6.3) Client selection between preferred address families and
  concurrent client-migration interaction — client behavior.
- (§9.7) IPv6 flow label generation — socket/OS-layer concern outside the
  transport SDK.
- (§10.1.1, §10.1.2) Liveness testing and idle-timeout deferral guidance —
  application policy (keep-alive helpers exist but the RFC imposes no
  normative wire behavior).
- (§14.2.1, §14.3.3) ICMP PTB message processing — the SDK deliberately
  ships probe-based DPLPMTUD only, without an ICMP channel.
- (§21.2–§21.14) Security-considerations deployment mitigations (Slowloris
  limits, request-forgery ingress filtering, stateless-reset-oracle
  cluster routing, traffic-analysis padding policy) — operator/application
  concerns.
- (§22) IANA registry policies — no implementation behavior.
