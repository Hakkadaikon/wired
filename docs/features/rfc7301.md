# RFC 7301 — TLS Application-Layer Protocol Negotiation (ALPN)

EARS requirement ledger extracted from the spec text
(`tasks/specs/rfc7301.txt`, not in git). Each requirement carries the test
that demonstrates it; an unchecked box with no test line is an open gap.
This ledger is scoped to the server side of ALPN, matching wired's role.
Status as of 2026-07.

Legend:

- `[x]` — demonstrated by the referenced test
- `[~]` — exercised indirectly (evidence line explains how; no dedicated test)
- `[ ]` — not demonstrated by any test yet

**Coverage: 13/14 tested, 1 indirect, 0 untested.**

## §3.1 The Application-Layer Protocol Negotiation Extension

- [x] 7301-001 (§3.1) The server shall parse a ClientHello
  "application_layer_protocol_negotiation(16)" extension whose
  extension_data is a ProtocolNameList: a 2-byte list length followed by
  one or more 1-byte-length-prefixed protocol names.
  - test: `tests/tls/alpn_test.c` — `test_alpn_h3_roundtrip`
  - test: `tests/tls/negotiate_test.c` —
    `test_negotiate_selects_h3_from_clienthello`
- [x] 7301-002 (§3.1) The server shall locate the ALPN extension
  (extension_type 0x0010) inside a ClientHello's extensions block by its
  type.
  - test: `tests/tls/ch_ext_test.c` — `test_ch_ext_finds_alpn_and_sni`
- [x] 7301-003 (§3.1) Protocol names in the ProtocolNameList are
  non-empty, IANA-registered opaque byte strings; empty strings must not
  be included and byte strings must not be truncated.
  - test: `tests/tls/alpn_test.c` — `test_alpn_encode_guard`
  - test: `tests/tls/alpn_test.c` — `test_alpn_bad_name_len`
- [x] 7301-004 (§3.1) Servers that receive a ClientHello containing the
  ALPN extension may return a protocol selection response to the client,
  ignoring any protocol name it does not recognize.
  - test: `tests/tls/negotiate_test.c` — `test_negotiate_rejects_non_h3`
  - test: `tests/tls/negotiate_test.c` — `test_negotiate_h3_among_others`
- [x] 7301-005 (§3.1) A ServerHello ALPN extension_data, when sent, is
  structured the same as the client's but must contain exactly one
  ProtocolName.
  - test: `tests/tls/negotiate_test.c` — `test_negotiate_build_response`
  - test: `tests/tls/eebuild_test.c` — `test_eebuild`
- [~] 7301-006 (§3.1) Unlike other TLS extensions, ALPN establishes a
  property of the connection, not the session: on session resumption the
  previous contents of this extension are irrelevant and only the values
  in the new handshake messages are considered.
  - evidence: the server's ALPN negotiation (`sdrv_negotiate_alpn` in
    `src/tls/handshake/core/sdrv/sdrv.c`) re-derives `s->alpn` from each
    ClientHello's own ALPN extension unconditionally, including on a
    resumption handshake; there is no code path that reuses a prior
    session's ALPN choice — no dedicated resumption+ALPN test.

## §3.2 Protocol Selection

- [x] 7301-007 (§3.2) The server should select the most highly preferred
  protocol that it supports and that is also advertised by the client.
  - test: `tests/tls/negotiate_test.c` — `test_negotiate_prefers_h3_over_hq`
- [x] 7301-008 (§3.2) In the event that the server supports no protocol
  that the client advertises, the server shall respond with a fatal
  "no_application_protocol" alert.
  - test: `tests/tls/alpnver_test.c` — `test_alpnver_require`
  - test: `tests/tls/negotiate_test.c` —
    `test_negotiate_none_when_neither_offered`
  - gap: `quic_alpnver_require` reports "no protocol selected" for the
    caller to fail the handshake; no `no_application_protocol` alert byte
    is itself emitted anywhere in `src/`.
- [x] 7301-009 (§3.2) The protocol identified in the ALPN extension in the
  ServerHello shall be definitive for the connection, until renegotiated;
  the server shall not respond with a selected protocol and subsequently
  use a different protocol.
  - test: `tests/tls/eebuild_test.c` — `test_eebuild_rejects_no_negotiation`
  - evidence: `quic_salpn_choice` is decided once by
    `sdrv_negotiate_alpn` and the same value is what
    `quic_eebuild_encrypted_extensions` echoes into EncryptedExtensions;
    QUIC has no renegotiation (RFC 9001), so "until renegotiated" is
    vacuously true.

## Protocol name registry (§6 IANA Considerations)

- [x] 7301-010 (§6) The "h3" protocol identifier (this server's ALPN
  choice for HTTP/3, RFC 9114) shall be matched by exact byte comparison
  (0x68 0x33).
  - test: `tests/tls/alpn_match_test.c` — `test_alpn_match_is_h3`
- [x] 7301-011 (§6) Protocol name comparison shall be exact and
  length-sensitive: a name that is a prefix or extension of a registered
  identifier shall not match.
  - test: `tests/tls/alpn_match_test.c` — `test_alpn_match_is_h3`
  - test: `tests/tls/alpn_match_test.c` — `test_alpn_match_equal`

## Wire-format robustness (implicit in §3.1's ProtocolNameList grammar)

- [x] 7301-012 The server shall reject an ALPN extension whose list-length
  or name-length field overruns the available bytes rather than
  over-reading.
  - test: `tests/tls/alpn_test.c` — `test_alpn_truncated`
  - test: `tests/tls/alpn_test.c` — `test_alpn_bad_name_len`
  - test: `tests/tls/ch_ext_test.c` — `test_ch_ext_truncated_returns_zero`
- [x] 7301-013 The server shall encode a ProtocolNameList entry only when
  it fits the destination buffer and the protocol name is non-empty and
  at most 255 octets.
  - test: `tests/tls/alpn_test.c` — `test_alpn_encode_guard`
- [x] 7301-014 The server shall build the EncryptedExtensions ALPN
  response consistently for either supported protocol ("h3" or
  "hq-interop", the interop runner's non-IETF HTTP/0.9-over-QUIC id) and
  must fail closed (build nothing) when no protocol was negotiated.
  - test: `tests/tls/eebuild_test.c` — `test_eebuild_selects_hq`
  - test: `tests/tls/eebuild_test.c` — `test_eebuild_rejects_no_negotiation`
